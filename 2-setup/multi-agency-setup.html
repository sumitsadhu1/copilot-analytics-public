0. Control Boundary and Go-Live Gates

Viva Insights partitions are controls for analyst query workspaces: they determine the rows and columns available to an assigned analyst in a partition. They do not, by themselves, govern the Copilot Dashboard, Analytics Reports, Agent Dashboard, metrics export, delegation, manager access, privileged roles, the global partition, downloaded results, or Power BI. Restricted Reporting is therefore a defense-in-depth reporting-segmentation pattern, not cryptographic, tenant, product-security, or certified compliance isolation. See Partitions in Viva Insights (last updated 23 June 2025).

Supported Environment and Licensing Gate

Do Not Proceed on Plan-Level Assumptions

Availability is feature- and cloud-specific. Supported Microsoft 365 environments for Viva Insights features (30 June 2026) says GCC does not support Agent Dashboard, Copilot metrics export, or Copilot Power BI reports. Connect to the Agent Dashboard (preview) (16 June 2026) says Agent Dashboard is unavailable in national and regional clouds. Base plan support in Environment requirements for Viva Insights (12 May 2026) does not establish feature parity. Record the exact cloud and verify every required surface in the tenant UI and Message Center before design.

For advanced access, Microsoft pages describe the threshold differently. The environment article explicitly gives 30 Microsoft 365 Copilot + 20 Viva Insights as 50 assigned Viva Insights service plans; the Dashboard article describes 50 Copilot or 50 Viva Insights. Do not state that mixed licence assignments fail. Verify assigned service plans, feature presence, and processing state. For Agent Dashboard, use the conservative feature-specific prerequisite of 50 Microsoft 365 Copilot licences plus agent activity; a 50-Viva-Insights-only tenant must validate availability. Escalate a documented mismatch to Microsoft Support.

Organizational Data Source and Precedence Gate

Record Attribute Ownership Before Upload

Do not describe an upload as universally or permanently replacing Entra. Uploaded data changes the source for supplied dashboard attributes; parallel ingestion can keep or restore Entra for ManagerId and Organization; Microsoft 365 Profile precedence is separate. Before change, record each required attribute, current source, owner, effective date, coverage, and target surface in Data Quality and Profile. Check Setup → Migration and imports, Message Center, and the actual tenant UI. Use Organizational Data in Microsoft 365 for new or edited uploads unless the tenant explicitly exposes a supported legacy path; use documented update/delete behavior, never a dummy-row workaround.

Source basis: Use your organizational data in Microsoft 365 and Microsoft Viva (2 December 2025); Upload and maintain data through the Microsoft 365 admin center (5 December 2025); Use Microsoft Entra plus .csv files for parallel data uploads (30 June 2025); and Import and manage your organizational data with a .csv file (2 April 2026). Workday import has a documented 100,000-user limit (Import organizational data from Workday, 6 April 2026), and API-based import remains preview (Import organizational data using API-based import (preview), 6 April 2026).

Change Control, Negative Tests, and Evidence

  1. Before change: export or capture VFAM policies, access lists, role assignments, delegations, manager hierarchy, partition filters and analysts, org-data source/quality, query assets, and Power BI workspace/app/RLS/export settings. Define owner, maintenance window, approvers, success/failure criteria, and restoration sequence.
  2. Test cohort first: use representative agency, shared-services, cross-agency, and privileged accounts. Treat enabling partitions as a high-impact tenant change; turning partitions off requires Microsoft Support. Export query assets before enabling, and confirm deletion impact before removing a partition.
  3. Mandatory actors: agency leader, cross-agency manager, shared executive, delegate, agency analyst, multi-partition analyst, global-partition analyst, report Viewer, Power BI Member/Contributor, AI Administrator, Insights Administrator, Insights Analyst, and Global Administrator.
  4. Mandatory surfaces: Copilot Dashboard, Analytics Reports, Agent Dashboard, weekly/daily metrics export, advanced query workspace, query-result download, delegation, manager-published reports, Power BI workspace/app, report export, Analyze in Excel, subscriptions, and resharing.
  5. Evidence: capture test date, tester identity/role, group/partition membership, policy export, org-data snapshot, expected denial or allowed result, observed result, screenshot/audit record, Power BI permissions, owner, and remediation ticket.
  6. Publication and go-live veto: any failed or unexecutable negative test blocks publication and production rollout. Roll back the change or remove access, preserve evidence, and open Microsoft Support when documented and observed behavior differ.
  7. Approval: obtain written Security, Compliance/Legal, Privacy/DPO, Viva Insights owner, and Power BI owner sign-off on tested behavior and residual risks. Product documentation is not a compliance attestation.
  8. Retest: run the full suite quarterly and after any role, hierarchy, delegation, org-data, partition, VFAM, licence, cloud, preview-status, or Power BI sharing change.

1. Default Priority Pattern Setup

Estimated time: 2–4 hours  |  PowerShell required: No

This author-created pattern relies on native Copilot Dashboard scoping (“Your group”) driven by Entra ID manager hierarchy. An agency CEO can receive that scope only when access, tenant, and reporting-line conditions are met (see callout below). Cross-agency and tenant-wide visibility from other access sources is explicitly accepted in this model.

Automatic Access Has Hard Prerequisites

Per Microsoft Learn, automatic dashboard access for senior leaders is granted only when both tenant criteria are met:

  • The tenant has at least 2,500 users with an Entra ID; and
  • The majority of users share a single reporting line (e.g., report up to the same CEO).

Within a qualifying tenant, a leader is auto-identified only if they are in the top three levels with a significant portion of the org reporting to them. A separate agency’s top leader is auto-identified only if that agency’s hierarchy tree is at least 4% of the overall company size. Consequences for multi-agency tenants:

  • Tenant under 2,500 users → no agency CEO gets automatic access. Use manual access grants + delegation (Phase 2).
  • Small agency (<4% of the tenant) → its CEO is not auto-identified, even in a large tenant. Grant access manually.

The “Your group” scope still works once access is granted manually — these gates govern automatic identification only. Access is re-evaluated weekly.

Phase 0: Base Setup

Complete all five steps in this guide’s implementation framework in the Admin Setup Guide before proceeding. The five-step sequence is an author-created implementation aid, not Microsoft’s capability taxonomy. This establishes:

  • Copilot and Viva Insights service-plan assignment using the conflict-safe licensing wording in Section 0; Agent Dashboard has a separate conservative preview prerequisite
  • Insights Admin and Insights Analyst roles
  • VFAM feature access policies (dashboard, advanced analytics, behavioural experiences)
  • Org data upload (Entra connection or CSV)
  • Default partition with analyst assignment
Verify Before Continuing

Run through the Global Admin Readiness Checklist in the Admin Setup Guide. All 8 checks must pass before you proceed to Phase 1.

Phase 1: Verify AutoCxO and Delegation

StepActionWhereExpected Result
1.1 Confirm Auto CxO Identification is on On by default — there is no web-app “General” toggle. If it was previously disabled, an AI Administrator re-enables it via the AutoCxoIdentification VFAM policy in PowerShell (tenant-level only). See MS Learn. Auto-enablement is active. Qualifying senior leaders are identified weekly from the Entra ID manager hierarchy (subject to the prerequisites above).
1.2 Confirm delegation is available No admin “enable” step exists. Delegation is self-service for eligible Dashboard users once the tenant meets the applicable 50-assigned-service-plan gate. Apply the source-conflict and tenant-validation rule in Section 0; do not reject a mixed assignment by shorthand. See Delegate access to the Copilot Dashboard, Agent Dashboard, and Copilot insights (last updated 9 February 2026). Leaders with access can delegate to others from Copilot Dashboard → Settings → Delegation.
1.3 Verify each agency CEO gets automatic “Your group” scope Ask each agency CEO to open the Copilot Dashboard and check the scope selector (top of dashboard) Each CEO sees “Your group” scoped to their direct + indirect reports. The population reflects Copilot-licensed employees only, and groups below the minimum group size (default 10) show “--” — so the count tracks licensed headcount, not raw agency headcount.
If a CEO Doesn’t See “Your Group”

Check these causes in order:

  1. Automatic-access prerequisites not met — tenant under 2,500 users, no single majority reporting line, or the agency tree is under 4% of the tenant (see the warning at the top of this section). In these cases the CEO is never auto-identified; grant access manually and use delegation.
  2. Incorrect or missing ManagerId data in Entra ID — verify the CEO’s reporting chain: every employee in the agency must have a valid manager chain that terminates at the agency CEO. See the Data Onboarding Guide for manager hierarchy verification steps.
  3. Licence or processing delay — data processing can take up to seven days after licences are assigned, and access is re-evaluated weekly.

Phase 2: Grant Additional Access

StepActionWhereExpected Result
2.1 Delegate to agency CIOs / senior leaders Each agency CEO uses the Delegation feature (Copilot Dashboard → Settings → Delegation) to grant access to their CIO or other leaders Delegated leaders see the same “Your group” scope as the delegating CEO.
2.2 Grant Central IT “Your company” scope A senior leader who already holds “Your company” scope delegates to the Central IT lead via Copilot Dashboard → Settings → Delegation (delegation conveys the delegator’s scope). If no qualifying senior leader exists, add the Central IT lead through Manage access to the Viva Insights web app (admin access list). Central IT lead sees “Your company” scope with tenant-wide data for cross-agency oversight.

Phase 2.3: Enrich Org Data (Optional)

If the Entra ID connection provides sufficient attributes (ManagerId, Organization), no additional data upload is needed. If you require custom attributes (e.g., Agency, FunctionType, CostCentre), upload a CSV supplement:

  1. Enable the Entra connection for ManagerId + Organization attributes (if not already done in Phase 0).
  2. If custom attributes are needed, prepare a CSV file following the CSV Templates (Section 4).
  3. Upload via the steps in the Data Onboarding Guide.

Phase 3: Create Partitions (Optional)

Partitions are optional in the Default Priority pattern. Use them when you want the rows and columns available in an analyst query workspace scoped to an agency; they do not scope the other surfaces listed in the boundary matrix.

  1. Navigate to Viva Insights web app → Admin → Privacy settings → Partitions.
  2. Click New partition.
  3. Name the partition after the agency (e.g., “Agency A”).
  4. Set the filter to the relevant attribute (e.g., Organization = "Agency A" or Agency = "Agency A").
  5. Assign the partition’s analyst(s).
  6. Repeat for each agency.

Phase 4: Power BI Reports (Optional)

For deeper analysis beyond the native dashboard, analysts can create Power BI reports from partition-scoped queries.

  1. The assigned analyst opens their partition in the Viva Insights web app.
  2. Creates a Copilot person query (or uses a pre-built template).
  3. Connects the query output to Power BI Desktop.
  4. Publishes the report to a Power BI workspace dedicated to the agency.
  5. Shares the workspace with agency leadership.

For step-by-step guidance on publishing to the Power BI service and assigning stakeholders by workspace role (Viewer, Member, etc.) — plus the per-viewer licensing implications — see Analyst Playbook — Share via Power BI (§5.2).

Default Priority Setup Complete

Configuration is complete only after the expected leader and delegate scopes, optional analyst query workspaces, and Power BI permissions pass the negative-test and evidence gate in Section 0. Continue to Ongoing Governance.

2. Restricted Reporting Pattern Setup

Estimated time: 1–3 days  |  PowerShell required: Yes (group-scoped VFAM restrictions; partitions are created via the Viva Insights web app UI)

This author-created pattern combines group-scoped dashboard restrictions, partition-scoped analyst query workspaces, controlled privileged access, and dedicated Power BI delivery. A partition limits rows and columns in its assigned analyst workspace; it does not guarantee what that person can access through Dashboard, Agent, export, delegation, manager, global, administrative, downloaded-result, or Power BI surfaces. The design is defense in depth and reporting segmentation, not tenant or security isolation.

Phase 0: Prerequisites

Complete through Tier 2, Step 3 in the Admin Setup Guide. This establishes:

  • Copilot and Viva Insights service-plan assignment using the conflict-safe licensing wording in Section 0
  • Insights Admin and Insights Analyst roles
  • VFAM verification that the dashboard toggle and Advanced Analytics toggle are accessible
Do NOT Enable Broad Access

Stop after verifying VFAM in Tier 2, Step 3. Do not enable AutoCxO, delegation, or broad dashboard access for “Everyone”. Restricted Reporting applies group-scoped controls in Phase 1 and then verifies each surface independently.

Phase 1: Group-Scoped VFAM Restrictions

Target direct Viva Insights web-app and Copilot Dashboard access to the approved Central IT population, while agency leaders receive reports through separately governed Power BI delivery. This is one layer of the pattern and must be tested alongside all other surfaces.

Important: Dashboard Access Is Now Managed via the Viva Insights Web App Toggle

Per Manage settings for the Microsoft Copilot Dashboard, Agent Dashboard, and Viva Insights web app (last updated 20 March 2026), the target state uses the Viva Insights web app control; pre-rollout tenants may still display the legacy Copilot Dashboard name, and existing policies carry forward. Disabling the web-app control removes the targeted user’s access to the dashboard, Copilot Analytics Reports, Agent Dashboard, and advanced analyst tools. Confirm the actual tenant label and effective policy before relying on it.

StepActionDetails
1.1 Disable Auto CxO Identification Viva Insights web app → Admin → General → Auto CxO Identification → toggle Off (or run the AutoCxoIdentification VFAM disable policy)
1.2 Disable Delegation Viva Insights web app → Admin → Delegation → toggle Off (or run the CopilotDashboardDelegation VFAM disable policy)
1.3 Restrict admin-center access list to Central IT Admin Centre → Settings → Viva → Viva Insights → Manage access to the Viva Insights web app for individual users and groups → add the Central IT Entra group and remove all individual agency users
1.4 Create Entra security group Create a group named AgencyUsers-NoDashboard in Entra ID. Add all non-Central-IT users (or all agency users).
1.5 Apply group-scoped VFAM policy Run the PowerShell command below to disable the entire Viva Insights web app (which includes the Copilot Dashboard) for the agency users group.
PowerShell: Disable the Viva Insights Web App for Non-Central-IT Users

Run this in an elevated PowerShell session with ExchangeOnlineManagement module 3.5.1 or later installed (required for non-mail-enabled group support):

Add-VivaModuleFeaturePolicy `
  -ModuleId VivaInsights `
  -FeatureId VivaInsightsWebApp `
  -Name DisableForNonCentralIT `
  -IsFeatureEnabled $false `
  -GroupIds "group-id-here"

Replace group-id-here with the Object ID of the AgencyUsers-NoDashboard Entra security group. Changes take effect within 24 hours.

Note: The deprecated -FeatureId CopilotDashboard command no longer applies a policy — the feature was removed when access management moved to the Viva Insights web app toggle.

Why Group-Scoped — Not “Everyone”

Use group-scoped policies, not “Everyone”. A blanket “Everyone” disable would also block Central IT and non-agency executives who may legitimately need dashboard access. The group-scoped approach protects these users while locking out agency-level browsing. Because the VivaInsightsWebApp toggle disables the entire web app for the scoped users (dashboard, Analytics Reports, Agent Dashboard, and analyst tools), confirm your Central IT analysts and any executives who need direct dashboard access are explicitly added to the access list in Step 1.3 before applying this policy.

Choose your data path based on whether your agencies have distinct email domains or you need a custom attribute for agency identification.

Path A: CSV with Agency Attribute

  1. Prepare a CSV file with the following columns (see Section 4 for a template):
    • Microsoft_PersonEmail (required)
    • Microsoft_ManagerEmail (required)
    • Microsoft_Organization (required)
    • EffectiveDate (required)
    • Agency (custom attribute — this is the partition filter)
  2. Follow the upload steps in the Data Onboarding Guide.
  3. After upload, verify the Agency attribute appears in Viva Insights → Admin → Data quality.

Path B: Entra Domain Filter (Zero CSV)

  1. No data upload is required.
  2. Verify each agency has a distinct email domain in Entra ID (e.g., @agencyA.gov, @agencyB.gov).
  3. The Domain attribute is system-generated from user email addresses and is automatically available for partition filtering.

Phase 3: Create Partitions

Create one partition per agency. Optionally, create a global partition for CIO-level cross-agency reporting.

StepActionDetails
3.1 Navigate to partitions Viva Insights web app → Admin → Privacy settings → Partitions
3.2 Create agency partition Click New partition. Name it after the agency (e.g., “Agency A”). Set the filter:
  • Path A: Agency = "Agency A"
  • Path B: Domain = "agencyA.gov"
3.3 Assign analyst(s) Assign the per-agency analyst to the partition. In that partition’s analyst query workspace, available rows and columns are scoped by the partition filter. Verify the analyst has no unintended global or multi-partition assignment and test their other access surfaces separately.
3.4 Repeat for each agency Create separate partitions for Agency B, Agency C, etc., each with its own filter and analyst(s).
3.5 Create global partition (optional) For CIO-level reporting, create a partition with no filter (includes all users). Assign the Central IT analyst.

Phase 4: Report Delivery

In Restricted Reporting, the recommended agency-facing delivery path uses Power BI workspaces/apps rather than direct Copilot Dashboard access. Power BI permissions are a separate control plane and require their own least-privilege design and negative tests.

  1. Each per-agency analyst opens their partition in the Viva Insights web app.
  2. Creates a Copilot person query (or uses a pre-built template such as the M365 Copilot Adoption report).
  3. Connects the query output to Power BI Desktop.
  4. Publishes the report to a dedicated Power BI workspace for the agency (e.g., “Agency A — Copilot Analytics”).
  5. Shares the workspace with agency leadership (CEO, CIO, designated leaders).
  6. Repeat for each agency.

For the detailed publish-and-assign steps (Power BI service, workspace roles such as Viewer, Entra-group access, and per-viewer Pro/PPU/Premium licensing), see Analyst Playbook — Share via Power BI (§5.2).

WARNING: Manager Publishing Can Broaden Access

“Publish Reports” with a manager hierarchy filter can make enabled managers eligible for additional Viva Insights experiences. A shared executive or cross-agency manager hierarchy may span agencies. Use dedicated Power BI workspaces/apps as the primary agency delivery path and test manager publishing, Dashboard scope, export, and delegation before any exception.

Restricted Reporting Configuration Complete

Configuration is not approved for publication or go-live until every required actor and surface passes the negative-test suite, evidence is retained, residual risks are signed off, and rollback is ready. Continue to Ongoing Governance.

3. Ongoing Governance

Multi-agency configurations require ongoing attention. Use the table below as a maintenance schedule.

FrequencyTaskArchitectureDetails
Monthly Refresh org data Both (Path A only for Restricted Reporting) Re-upload CSV with current employee data. Path B (Domain filter) requires no action.
Monthly Verify partition filters Both (if using partitions) Spot-check partition member counts in Viva Insights → Admin → Partitions. Counts should match expected agency headcount.
Monthly Spot-check dashboard scope Default Priority Have one leader per agency verify their “Your group” scope shows the correct population.
Quarterly Audit delegation list Default Priority Review who has been delegated access in Viva Insights → Admin → Delegation. Remove departed staff.
Quarterly Repeat full boundary test and access recertification Restricted Reporting Re-run all actor/surface negative tests; recertify global, multi-partition, shared-executive, cross-agency-manager, delegation, VFAM, Agent/export, and Power BI access. A failure vetoes continued publication until remediated.
Quarterly Review access logs in Purview Both Check Microsoft Purview audit logs for dashboard access events. Verify no unexpected users accessed analytics.
On org change Update partitions Both (if using partitions) Use approved change control for agency merge, split, rename, shared-services reassignment, or analyst conflict-of-interest changes. Capture pre-change state, update filters/assignments, retest every surface, and retain evidence.
On org change Update CSV Path A (Restricted Reporting) Re-upload CSV immediately when new hires join, employees transfer between agencies, or agencies restructure.
