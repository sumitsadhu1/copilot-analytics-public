1. Privacy Thresholds

Privacy thresholds protect aggregated organization insights; they do not turn every Viva Insights output into aggregate-only data. Insights Analysts can receive de-identified row-level employee records in advanced analysis, and uploaded organizational attributes can make a person inferable when combined.

SettingDefaultConfigurable Range
Minimum group size10 users5–unlimited (lowest allowed = 5)

How Thresholds Work

  • Data suppression: Results for groups below the threshold are suppressed — they show as blank or “insufficient data” rather than displaying potentially identifiable information.
  • Dashboard and manager/leader insights: These surfaces use protected aggregates, minimum group size, distribution masking, and differential privacy where documented.
  • Advanced analysis: An Insights Analyst can query de-identified individual-level organizational and Microsoft 365 metrics. The encrypted person identifier is not a guarantee of anonymity if title, location, organization, or custom attributes enable inference.
  • Access governance: Treat analyst output as potentially personal data. Use least-privilege partitions, require role-specific privacy/security training, define an approved analysis purpose, and audit authored queries against policy and any data protection impact assessment.

Sources: Advanced analysis privacy — Personal data, de-identified data, and calculated data (Microsoft Learn, updated 10 December 2025) documents de-identified row-level output and re-identification risk; Roles in Viva Insights — Levels of responsibility (7 July 2026) distinguishes analyst row-level access; and Advanced analysis privacy — Advanced analysis access (10 December 2025) recommends training and states that analyst-authored queries are logged for audit.

Minimum Team Size vs Minimum Group Size — They Are Different

These two settings apply in different contexts and are configured in different places. Both have a minimum of 5.

SettingWhere ConfiguredWhat It ControlsWho It Counts
Minimum team size Viva Insights → Admin → Manager settings Manager-facing insights (My Team tab, Copilot Dashboard manager view) Includes the manager in the headcount
Minimum group size Viva Insights → Admin → Privacy settings Copilot Dashboard aggregated views, Power BI query results Does not include the viewer in the headcount

Example: If both are set to 5, a manager with exactly 4 direct reports will see their team data (4 reports + 1 manager = 5, meeting the team size threshold), but an analyst filtering a dashboard to a group of 4 users will see suppressed data (4 < 5, below the group size threshold).

2. Exclusion Lists

The unified exclusion list is a reporting control, not a universal statement that data is never collected or processed. It tells the documented Dashboard and Viva reporting surfaces not to show specified users in their aggregates.

What Gets Excluded

  • Microsoft Copilot Dashboard and Agent Dashboard aggregated insights
  • Viva Insights behavioral collaboration reports in advanced analysis
  • Microsoft 365 Copilot Chat (web) usage in the Copilot Dashboard, including a Microsoft 365 licensed user who does not have a Copilot or Viva Insights license

Microsoft Learn does not describe this list as deleting source data or excluding a person from every query type and processing path. For exclusion from advanced-analysis processing, manage the user’s Viva Insights service plan as part of the data-subject-request workflow below.

How to Upload

Microsoft Entra groups Create or select an Entra group containing users to exclude, then configure the unified exclusion list through the current Microsoft 365 admin center workflow.

Processing Times

Plan Exclusions Before Dashboard Launch

Because exclusion-list processing can take up to five days, configure and verify the list before launch. Confirm the policy is on, inspect the resulting membership, and test the affected reports after the processing window.

Source: Manage settings for the Microsoft Copilot Dashboard, Agent Dashboard, and Viva Insights web app — unified exclusion list (Microsoft Learn, updated 20 March 2026).

3. Data Residency

Microsoft documents local data-at-rest behavior for Viva Insights advanced analysis, manager, and leader collaboration activities and metric computations. The location is determined by the tenant’s default geography (default mailbox region), not by each user’s location. It is not a universal statement about every Copilot Analytics workload.

Viva Insights provisioned afterTenant default mailbox regionAdvanced, manager, and leader data stored at rest in
4 June 2024Australia (AUS)Australia
12 August 2024Japan (JPN)Japan
13 September 2024Canada (CA)Canada
13 September 2024United Kingdom (GB)United Kingdom

Customers provisioned before the applicable date continue to be hosted in the region used before local residency became available. Use Microsoft’s static workload-location table for that case rather than inferring location from the current tenant geography.

Verification Steps

  1. Record the tenant default mailbox region and the date Viva Insights was provisioned.
  2. Apply the matrix above. For earlier provisioning, consult the linked static workload-location information.
  3. If either value is unavailable or the result is material to a residency attestation, obtain tenant-specific confirmation through Microsoft Support; do not infer it from a user mailbox or a generic tenant-region label.
  4. Have legal/compliance approve the documented workload scope and evidence before making a sovereignty statement.

Source: Viva Insights data residency for advanced analysis, managers, and leaders (Microsoft Learn, updated 23 June 2025), including its link to static data location information for select workloads.

3B. Microsoft Purview DLP for Copilot Prompts

Microsoft Purview Data Loss Prevention (DLP) uses the Microsoft 365 Copilot and Copilot Chat policy location. The actions below have different effects and must not be described as one generic prompt block.

Conditions, Actions, and Status

Condition and exact actionStatusDocumented result
Content contains > Sensitive information types
Prevent Copilot from processing content > Performing Web Searches		Supported action; the article does not label this action previewBlocks external web search as a grounding source. Copilot can still respond using permitted internal Microsoft 365 data.
Content contains > Sensitive information types
Prevent Copilot from processing content > Processing prompts		Preview; rolling out to tenantsCopilot does not respond, and the prompt is not used for internal or web searches. Check whether rollout has reached the tenant. Applies to Microsoft 365 Copilot, Copilot Chat, Word, Excel, PowerPoint, and prebuilt agents.
Content contains > Sensitivity labels
Prevent Copilot from processing content		Generally availableSelected labeled files and emails are not processed or used in the response summary, although an item can still appear in citations. Calendar invites are unsupported; documented email coverage starts 1 January 2025.
Prompt SIT Blocking Is Preview

Do not treat Processing prompts as a guaranteed production boundary until the tenant exposes the preview control and a positive/negative test confirms enforcement. During preview, Word, Excel, and PowerPoint messaging might not clearly identify organizational policy as the reason for the block.

Important Limitations

  • DLP evaluates text typed directly into a prompt; it cannot scan the contents of files uploaded directly in prompts.
  • Sensitive-information-type and sensitivity-label conditions cannot be combined in the same rule, although separate rules can be placed in one policy.
  • The location is available only in a Custom policy template; DLP alerts, notifications, and simulation mode are supported.
  • Policy updates can take up to four hours to appear in Copilot experiences.

Why This Matters for Analytics Teams

  • Analytics-count impact is not verifiable from Microsoft Learn. The DLP article documents response and search behavior, but does not state whether a blocked attempt is logged or counted in Copilot Analytics.
  • Do not use DLP as an explanation for adoption gaps without a controlled tenant test or Microsoft Support confirmation that covers the specific report and metric.
  • Coordinate Security/Compliance and Analytics teams so policy tests, dates, affected users, and observed report behavior are recorded together.

Validate the Unknown Analytics-Count Impact

Microsoft Learn does not document whether a DLP-blocked attempt appears in each Copilot Analytics metric. Do not infer the answer from the DLP alert or from one dashboard total. Use this controlled protocol when the distinction is operationally material:

  1. Approve the test: use a non-production or approved pilot cohort, synthetic values only, one documented DLP policy/action, and a fixed UTC test window.
  2. Establish controls: from the same app and test identity, submit a benign prompt that should succeed and a synthetic matching prompt that should be restricted. Record exact timestamps and expected outcomes.
  3. Prove policy behavior: retain the DLP simulation/alert record and the observed Copilot response for both prompts. This proves enforcement, not analytics counting.
  4. Wait for each report's documented latency: inspect the relevant Copilot Dashboard metric, admin-center report, export, or advanced query using the same population and time boundary. Do not compare different report windows.
  5. Record the result as tenant-specific: document whether the successful and blocked attempts can be distinguished, the report/metric/version, policy action, latency, and any ambiguity.
  6. Escalate ambiguity: if the result affects adoption reporting or a customer statement, open Microsoft Support with policy, timestamps, alert IDs, report/export evidence, expected behavior, and observed behavior. Until confirmed, retain the wording Not verifiable from Microsoft Learn.

Configuration Reference

DLP for Copilot is configured in the Microsoft Purview portal, not in Viva Insights. Source: Microsoft Purview DLP for Microsoft 365 Copilot and Copilot Chat (Microsoft Learn, updated 10 June 2026).

Complementary to Analytics

DLP for Copilot and Copilot Analytics serve different purposes but are part of the same deployment posture. A well-governed Copilot deployment uses DLP to prevent sensitive data exposure, Copilot Analytics to measure adoption and impact, and Organisational Messages to drive enablement. Consider establishing a cross-functional team that includes Security (Purview), IT (Admin Centre), and Analytics (Viva Insights) stakeholders.

4. GDPR and Regional Compliance

Microsoft 365 Copilot Analytics is designed with GDPR compliance in mind. The following controls support regulatory requirements:

  • End-user opt-out: When enabled, this removes the user’s behavioral metrics from future row-level outputs. It does not apply to aggregated organization insights, meeting-query results, or row-level Copilot usage data.
  • Data minimisation: Dashboards and manager/leader insights use protected aggregates; advanced-analysis queries can contain de-identified individual rows. Limit attributes, rows, retention, and query purpose to what the approved analysis requires.
  • Org data handling: If you upload organisational data (CSV with attributes like Department, Agency, Location), ensure the upload process follows your organisation’s data handling policies. Do not include sensitive personal data in org data attributes.

Data Subject Request Control Path

Regional Considerations

Some regions may have additional requirements beyond GDPR (e.g., UK-GDPR, Australia Privacy Act). Work with your legal and compliance teams to confirm that your privacy threshold settings, exclusion lists, and org data handling meet local regulations.

5. Data Retention

Retention of backend Viva Insights data is different from a dashboard display window or an export range. Do not use one as proof of the other.

Data SourceRetention PeriodNotes
Advanced Analytics — Collaboration metrics Initially 13 months; grows weekly to a maximum of 27 months No collaboration data older than 27 months is retained.
Advanced Analytics — Copilot metrics Maximum 13 months Microsoft 365 Copilot metrics do not grow to the 27-month collaboration window.
Dashboard views, dashboard export, and Microsoft 365 admin usage reports Separate feature-specific windows Use the current documentation for the specific surface; these windows do not redefine the 27/13-month Viva Insights retention policy.
Govern Long-Term Snapshots Separately

If you retain exported or Power BI snapshots beyond a product’s interactive window, apply an approved business purpose, access controls, retention schedule, and deletion process. A local snapshot is a new governed copy.

Source: Advanced analysis privacy — Retaining sensitive data (Microsoft Learn, updated 10 December 2025).
