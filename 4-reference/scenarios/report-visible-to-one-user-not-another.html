Microsoft 365 Copilot Analytics

Scenario 02
“Why can I see the Microsoft 365 Copilot adoption report in the Copilot Dashboard but my colleague can’t?”
Typically reported byA user who opened a Copilot report and a colleague (same team, similar licence) sees nothing or a permission-denied page.
Surfaces involvedanalysis.insights.cloud.microsoft, M365 Admin Center (VFAM), Entra ID
Roles touchedGlobal Admin, Insights Admin, AI Admin, AutoCxO leaders, Delegated users, People-managers
Version1.0 • Published June 2026

Quick answer

Access to the Copilot Dashboard (and therefore the Reports surface inside it) is granted through five independent paths. A colleague who sees nothing simply matches none of them — even on an identical licence. The Reports surface never has its own permission model: it only shows reports to people who already have dashboard access through one of these paths.

#Access pathGranted toScope they see
1 Admin role Global Administrator, Insights Administrator, or AI Administrator Your company (whole tenant)
2 AutoCxO (automatic senior-leader identification) Senior leaders auto-detected from the org hierarchy Their own reporting hierarchy
3 Delegation Anyone a leader or admin explicitly shares access with Your group / Your company (whatever was delegated)
4 VFAM web-app grant Users/groups explicitly allowed the Viva Insights web app + Copilot Dashboard feature Per the access they otherwise hold
5 Manager Settings (“Grant managers scoped access”) Any people-manager whose team meets the minimum team size — once the Insights Admin toggles this on Your group (their own team)
The most common single cause

When two colleagues on the same licence differ, the explanation is usually path 5 (Manager Settings): the person who can see the report is a people-manager whose team meets the minimum team size; the colleague who can’t is an individual contributor (no direct reports), or their team is below the minimum, or their Manager attribute isn’t populated.

30-second diagnosis

  1. Is the colleague a people-manager? If they have no direct reports, paths 2 and 5 can’t apply to them. The difference is almost certainly that you have one of those and they don’t.
  2. Does the user who can see the report hold an admin role? Check myaccount.microsoft.com → My roles for Global / Insights / AI Administrator. If yes, that alone explains tenant-wide access — the colleague without the role sees nothing.
  3. Is Manager Settings enabled? analysis.insights.cloud.microsoft → Admin → Manager settings → confirm “Grant managers scoped access” is ON. If off, no manager sees team data regardless of licence.
  4. Is the colleague’s team below the minimum team size? Default minimum is 10 (configurable, floor 5) and includes the manager in the count. A manager of a 3-person team sees nothing.
  5. Is the colleague’s Manager attribute populated? No manager hierarchy in Entra ID (or uploaded org data) = no manager-scoped access.

Causes ranked by likelihood

#CauseWhy it happensFix & verification
1 You have manager-scoped access (path 5) and the colleague is an individual contributor. Once the Insights Admin enables “Grant managers scoped access” in Manager Settings, any people-manager whose team meets the minimum team size automatically gets scoped dashboard + Reports access to their own team. An IC with no direct reports never qualifies for this path.
  • Confirm you have direct reports and the colleague does not (org chart / Entra Manager attribute).
  • Verify: the scope chip on your dashboard reads “Your group” (your team), not “Your company.” An IC colleague would have no scope at all.
  • This is expected behaviour, not a bug — ICs are not meant to see team-level reports.
2 You hold an admin role (path 1); the colleague does not. Global Administrator, Insights Administrator, and AI Administrator all grant tenant-wide (“Your company”) access to the dashboard and Reports. A colleague without any of these roles has no admin-derived access.
  • Check both users at myaccount.microsoft.com → My roles.
  • If the colleague legitimately needs access, grant it via delegation (path 3) rather than handing out an admin role — least privilege.
  • Verify: your scope chip reads “Your company.”
3 Manager Settings is OFF tenant-wide. If the Insights Admin has never enabled “Grant managers scoped access,” then no manager sees team-level data — so if your colleague is a manager expecting their team report, this is why. (You would still see reports if your access comes from path 1–4.)
  • analysis.insights.cloud.microsoft → Admin → Manager settings → toggle “Grant managers scoped access” ON.
  • Allow ~1 hour for the change to propagate.
  • Verify: after propagation, a qualifying manager sees a “Your group” scope on the dashboard.
  • See Admin Setup Guide — Manager Settings step.
4 The colleague’s team is below the minimum team size. To protect individual privacy, manager-scoped views are suppressed unless the team meets the minimum team size (default 10, configurable down to a floor of 5; the manager counts toward the total). A manager of a small team sees an empty view even when everything else is configured correctly.
  • Count the colleague’s direct + indirect reports (including themselves).
  • If genuinely small, this is by design — the minimum cannot go below 5.
  • Note: minimum team size (Manager Settings) is distinct from minimum group size (Privacy settings) — see Privacy Configuration Guide.
5 The colleague’s Manager attribute (or manager hierarchy) isn’t populated. Manager-scoped access (path 5) and AutoCxO (path 2) both depend on a manager hierarchy — either the Entra ID Manager attribute or a ManagerId column in uploaded org data. Missing hierarchy means the platform can’t establish who reports to whom.
  • Check the colleague’s Entra profile (Manager field) and, if org data is uploaded, the ManagerId mapping.
  • Populate via HR sync, Entra, or the org-data CSV upload.
  • Verify: after the next data refresh (~1 hour for settings; longer for a full org-data reprocess), the manager view populates.
6 You received a delegation (path 3); the colleague did not. A leader or admin can delegate dashboard access to specific people (up to 75 delegates; no cascading). If you were delegated access and your colleague wasn’t, you see the reports and they don’t.
  • Ask whether a leader shared access with you. The delegating leader can add the colleague the same way.
  • Delegation rules: max 75 delegates, no cascading, scope follows the delegator; access is revoked if the delegator leaves the org.
  • See Dashboard Guide — “How to Delegate Access.”
7 A VFAM policy blocks the colleague’s web-app or Copilot Dashboard access (path 4). Viva Feature Access Management can disable the Viva Insights web app or the Copilot Dashboard feature for specific groups. If the colleague is in a group covered by a restrictive policy, the surface simply doesn’t load for them — even if they would otherwise qualify via paths 1–5.
  • admin.microsoft.com → Viva → Feature Access Management → Viva Insights → verify the Viva Insights web app and Copilot Dashboard features are ON for the colleague’s group.
  • Remember VFAM policies are group-scoped — confirm which policy applies to the colleague.
  • Verify: after the policy change, the colleague can load analysis.insights.cloud.microsoft.
8 The colleague lacks a Viva Insights / Copilot licence. Even with the right role or manager hierarchy, the web-app surface requires the appropriate licence assignment. An unlicensed (or differently licensed) colleague can be blocked at the door.
  • admin.microsoft.com → Users → Active Users → select the colleague → Licenses and apps → confirm the Viva Insights / Copilot entitlement.
  • Verify: the licence shows assigned and active.
9 Recent change hasn’t propagated yet. Role assignments refresh on next sign-in (up to ~60 min if a session stays open); Manager Settings changes take ~1 hour; org-data hierarchy changes can take longer to reprocess.
  • Have the colleague sign out of all M365 surfaces and back in (incognito is fastest).
  • Wait the full propagation window before escalating.
  • Verify: re-check myaccount.microsoft.com → My roles and the dashboard scope chip.
Most common misdiagnosis

This is usually escalated as “the same licence behaves differently for two people — it must be a bug.” In practice the two users almost always differ on (1) manager status or (2) admin role. Identical licences do not imply identical dashboard access — the licence is necessary but not sufficient.

Surface vs. scope — keep these separate

Two different questions are easy to conflate:

  • Does the dashboard / Reports surface appear at all? → governed by admin role, AutoCxO, delegation, and the VFAM web-app grant (paths 1–4).
  • Does a manager see team-level data within it? → governed by Manager Settings + manager hierarchy + minimum team size (path 5).

Diagnose which question you’re actually answering before chasing a fix.

What “done” looks like

  • You can articulate which of the five paths grants your access (admin role, AutoCxO, delegation, VFAM grant, or Manager Settings).
  • You’ve confirmed the colleague matches none of those paths — or identified the single one that’s missing and fixable (most often: enable Manager Settings, populate the Manager attribute, or add a delegation).
  • After the relevant change propagates, the colleague either sees the expected scoped report or it’s confirmed as correct-by-design (e.g. an IC with no team to report on).
