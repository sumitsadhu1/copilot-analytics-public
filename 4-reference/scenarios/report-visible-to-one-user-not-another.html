Microsoft 365 Copilot Analytics

Scenario 02
Who can see the Copilot Dashboard & its reports — and why two colleagues on the same licence differ
Typically reported byA user whose colleague (same team, similar licence) sees nothing or a permission-denied page — or an admin asking “who is supposed to see this, and how do I grant or restrict it?”
Surfaces involvedanalysis.insights.cloud.microsoft, M365 Admin Center (VFAM), Entra ID
Roles touchedGlobal Admin, Insights Admin, AI Admin, AutoCxO leaders, Delegated users, People-managers
Version1.0 • Published June 2026

Quick answer

The Reports surface has no permission model of its own — it only shows reports to people who already have Copilot Dashboard access. That access comes from three buckets, with a VFAM gate sitting on top that can block any of them. Crucially, most “why can’t my colleague see it?” cases are not a misconfiguration — the colleague is an individual contributor who was never meant to see team or company reports.

The whole model reduces to four questions: Am I in bucket 1 or 2? Am I licensed (bucket 3)? Is a VFAM gate blocking me? Every variant of “who can see this” — including “why me but not my colleague” — is answered by those four.

The most likely answer is “working as designed”

Before hunting for a broken setting: if your colleague is an individual contributor (no direct reports) and isn’t an admin or a delegate, they are correctly seeing nothing. The Copilot Dashboard reports on a group — an IC has no team to report on. Only treat this as a fault once you’ve confirmed the colleague genuinely should qualify via bucket 1 (automatic) or bucket 2 (delegation).

30-second diagnosis

  1. Should the colleague see it at all? If they’re an IC with no direct reports, no admin role, and no delegation — stop: this is working as designed. Nothing to fix.
  2. Which bucket grants your access? Check myaccount.microsoft.com → My roles (admin), whether you’re a people-manager (automatic), or whether a leader delegated to you (granted). That tells you exactly what the colleague is missing.
  3. Colleague is a manager but sees nothing? Confirm Manager Settings is ON (analysis.insights.cloud.microsoft → Admin → Manager settings → “Grant managers scoped access”), their team meets the minimum team size (default 10, floor 5, manager counts), and their Manager attribute is populated.
  4. Should qualify but still can’t load the surface? Check for a VFAM gate blocking their group, then confirm their Viva Insights / Copilot licence.

Causes ranked by likelihood

#CauseWhy it happensFix & verification
1 The colleague is an IC — not a manager, admin, or delegate. Working as designed. The Copilot Dashboard reports on a group (company, hierarchy, or team). An individual contributor with no direct reports, no admin role, and no delegation has nothing to report on, so the surface is intentionally empty for them. This is the single most common “why not my colleague” case — and there is nothing to fix.
  • Confirm the colleague has no direct reports (org chart / Entra Manager attribute), no admin role (myaccount.microsoft.com → My roles), and no delegation.
  • If all three are true, close the case as expected behaviour. If the colleague legitimately needs access, grant it via delegation (bucket 2) — don’t hand out an admin role.
2 You hold an admin role (bucket 1); the colleague does not. Global Administrator, Insights Administrator, and AI Administrator all grant tenant-wide (“Your company”) access. A colleague without any of these has no admin-derived access.
  • Check both users at myaccount.microsoft.com → My roles.
  • If the colleague needs access, prefer delegation over a new admin role (least privilege).
  • Verify: your scope chip reads “Your company.”
3 Manager Settings is OFF tenant-wide. If the Insights Admin has never enabled “Grant managers scoped access,” then no manager sees team-level data — so if your colleague is a manager expecting their team report, this is why. (You would still see reports if your access comes from an admin role or a delegation.)
  • analysis.insights.cloud.microsoft → Admin → Manager settings → toggle “Grant managers scoped access” ON.
  • Allow ~1 hour for the change to propagate.
  • Verify: after propagation, a qualifying manager sees a “Your group” scope on the dashboard.
  • See Admin Setup Guide — Manager Settings step.
4 The colleague’s team is below the minimum team size. To protect individual privacy, manager-scoped views are suppressed unless the team meets the minimum team size (default 10, configurable down to a floor of 5; the manager counts toward the total). A manager of a small team sees an empty view even when everything else is configured correctly.
  • Count the colleague’s direct + indirect reports (including themselves).
  • If genuinely small, this is by design — the minimum cannot go below 5.
  • Note: minimum team size (Manager Settings) is distinct from minimum group size (Privacy settings) — see Privacy Configuration Guide.
5 The colleague’s Manager attribute (or manager hierarchy) isn’t populated. Manager access and AutoCxO (both bucket 1) depend on a manager hierarchy — either the Entra ID Manager attribute or a ManagerId column in uploaded org data. Missing hierarchy means the platform can’t establish who reports to whom.
  • Check the colleague’s Entra profile (Manager field) and, if org data is uploaded, the ManagerId mapping.
  • Populate via HR sync, Entra, or the org-data CSV upload.
  • Verify: after the next data refresh (~1 hour for settings; longer for a full org-data reprocess), the manager view populates.
6 You received a delegation (bucket 2); the colleague did not. A leader or admin can delegate dashboard access to specific people (up to 75 delegates; no cascading). If you were delegated access and your colleague wasn’t, you see the reports and they don’t.
  • Ask whether a leader shared access with you. The delegating leader can add the colleague the same way.
  • Delegation rules: max 75 delegates, no cascading, scope follows the delegator; access is revoked if the delegator leaves the org.
  • See Dashboard Guide — “How to Delegate Access.”
7 A VFAM gate blocks the colleague’s web-app or Copilot Dashboard access. Viva Feature Access Management sits on top of every bucket and can disable the Viva Insights web app or the Copilot Dashboard feature for specific groups. If the colleague is in a group covered by a restrictive policy, the surface simply doesn’t load — even if they would otherwise qualify. (This may also be deliberate — see “Deliberately restricting access” below.)
  • admin.microsoft.com → Viva → Feature Access Management → Viva Insights → verify the Viva Insights web app and Copilot Dashboard features are ON for the colleague’s group.
  • Remember VFAM policies are group-scoped — confirm which policy applies, and whether the block is intentional before lifting it.
  • Verify: after the policy change, the colleague can load analysis.insights.cloud.microsoft.
8 The colleague lacks a Viva Insights / Copilot licence (bucket 3). Even with the right role or manager hierarchy, the web-app surface requires the appropriate licence assignment. An unlicensed (or differently licensed) colleague is blocked at the door.
  • admin.microsoft.com → Users → Active Users → select the colleague → Licenses and apps → confirm the Viva Insights / Copilot entitlement.
  • Verify: the licence shows assigned and active.
9 Recent change hasn’t propagated yet. Role assignments refresh on next sign-in (up to ~60 min if a session stays open); Manager Settings changes take ~1 hour; org-data hierarchy changes can take longer to reprocess.
  • Have the colleague sign out of all M365 surfaces and back in (incognito is fastest).
  • Wait the full propagation window before escalating.
  • Verify: re-check myaccount.microsoft.com → My roles and the dashboard scope chip.
Most common misdiagnosis

This is usually escalated as “the same licence behaves differently for two people — it must be a bug.” In practice the two users almost always differ on (1) manager / admin status — and very often the colleague is simply an IC seeing the surface correctly empty. Identical licences do not imply identical dashboard access — the licence is necessary but never sufficient.

Surface vs. scope — keep these separate

Two different questions are easy to conflate:

  • Does the dashboard / Reports surface appear at all? → governed by your bucket (admin role, AutoCxO, manager access, or delegation), your licence, and the VFAM gate.
  • Does a manager see team-level data within it? → governed by Manager Settings + manager hierarchy + minimum team size.

Diagnose which question you’re actually answering before chasing a fix.

Worked example — the executive sponsor who isn’t a manager

This case comes up constantly, so it’s worth walking through end-to-end because the intuitive answer is the wrong one.

The situation

James is the executive sponsor of the organisation’s Copilot rollout. He needs the org-wide adoption report to track the programme. He is not a people-manager — he has no direct reports — and he holds no admin role. The instinctive move is “he’s an exec, just give him manager access.” That doesn’t work, and it’s worth understanding why.

Why “manager” is the wrong lever. In Viva Insights, “manager” is not a statement about seniority or importance — it is a literal test of whether the person has direct or indirect reports, established by the PersonId / ManagerId hierarchy. You cannot enable someone as a manager who has nobody reporting to them, no matter how senior they are. And even if James did have a small team, manager access would only show that team’s adoption — not the company-wide view an executive sponsor actually needs.

The correct lever is delegation (bucket 2). A non-manager reaches the dashboard exactly one of these ways: automatic senior-leader identification (only if Entra ID places them at the top of the hierarchy — not James), an admin/analyst role (over-privileged for a read-only sponsor), Global Admin (far too much), or delegation. Delegation is the right answer twice over: it is the only mechanism open to a non-manager, and it is the only one that can carry a “Your company” scope — the org-wide adoption picture James is asking for.

StepAction
1 Identify a delegator who already has company-level access — a senior leader with automatic access, or a Global Administrator.
2 That leader opens the Copilot Dashboard → Settings → Delegation → Add delegates, adds James, and assigns the scope “Your company.”
3 Confirm James holds a Viva Insights / Copilot licence (bucket 3 — necessary even for delegates).
4 Check no VFAM gate blocks James’s group from the Viva Insights web app / Copilot Dashboard.
5 James opens the Viva Insights web app → Reports and sees the org-wide adoption report under the delegating leader’s view.
Delegation caveats to set expectations
  • A delegate sees the surface as the delegating leader — a banner shows whose insights they’re viewing.
  • Delegation is capped at 75 delegates per leader and cannot cascade (James can’t re-delegate).
  • If the delegating leader loses their own access (e.g. leaves the org), James’s access is revoked too — so prefer a stable senior leader or a Global Admin as the delegator.

This matches Microsoft Learn exactly: the Copilot Analytics reports prerequisites list automatic access, delegate access, Insights Analyst, or Global Admin — with no “promote a non-manager to manager” path — and the delegate-access guidance confirms a company-scoped delegation is how a non-manager gets the org-wide view.

Deliberately restricting access (the inverse problem)

The harder, higher-stakes version of this question is the reverse: “How do I make sure a particular leader or group cannot see the dashboard, even though their role or hierarchy would normally grant it?” This is the everyday reality in regulated, multi-agency, or shared-tenant deployments where one agency’s leaders must not see another’s data.

Buckets 1–3 are mostly automatic — you can’t easily stop the platform from auto-identifying a senior leader (AutoCxO) or a people-manager. The VFAM gate is the supported lever for removing access:

GoalHowWatch out for
Block a specific set of users from the dashboard Create a group-scoped VFAM policy that turns the Copilot Dashboard (and/or Viva Insights web app) feature OFF for an Entra group (e.g. AgencyUsers-NoDashboard). Scope the policy to the group, never to “Everyone” — an Everyone-level block also strips access from non-agency executives who legitimately need it.
Let agency managers see only their own agency Combine the VFAM block above with partition-based isolation so that, where access is granted, the data is already scoped to the agency. VFAM controls the surface; partitions control the data. You usually need both for true isolation.
Confirm a block actually took effect Have a test user in the blocked group sign out / in and load analysis.insights.cloud.microsoft. VFAM changes are not instant — allow propagation time before concluding it didn’t work.
Use group-scoped policies, never “Everyone”

The most common multi-agency mistake is blocking the Copilot Dashboard at the “Everyone” level to lock down agency users — which also blocks senior executives and central admins who are supposed to see tenant-wide adoption. Always target a dedicated Entra group (e.g. AgencyUsers-NoDashboard). See the Multi-Agency Isolation Architecture for the full group-scoped lockdown pattern.

What “done” looks like

  • You can name which bucket grants your access — automatic (admin role / AutoCxO / manager access), granted (delegation), plus a valid licence and no VFAM gate.
  • You’ve confirmed the colleague either matches no bucket — in which case it’s correct by design (most often an IC with no team) — or you’ve found the single missing, fixable item (enable Manager Settings, populate the Manager attribute, add a delegation, or lift a VFAM block).
  • If the goal was the inverse — restricting access — the right users are blocked via a group-scoped VFAM policy (never “Everyone”), verified by a test user in the blocked group.
