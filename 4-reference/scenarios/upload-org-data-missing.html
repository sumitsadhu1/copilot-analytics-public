Microsoft 365 Copilot Analytics

Scenario 01
“I can’t see the option to upload org data at aka.ms/copilotdashboard”
Typically reported byInsights Administrator who expected a CSV-upload button on the Copilot Dashboard.
Surfaces involvedM365 Admin Center, analysis.insights.cloud.microsoft, Entra ID
Roles touchedInsights Admin, Organizational Data Source Admin, AI Admin, Global Admin
Version1.1 • Last validated 16 July 2026

Quick answer

aka.ms/copilotdashboard redirects to the Copilot Dashboard view inside Viva Insights. That dashboard view is not the organizational-data ingestion UI. Determine the supported admin path from the tenant’s latest Message Center migration notice and the controls actually present in both admin surfaces; a missing tile or a calendar date is not sufficient evidence.

Official Microsoft Sources Conflict

Manage settings for the Microsoft Copilot Dashboard, Agent Dashboard, and Viva Insights web app (Microsoft Learn, updated 20 March 2026) still recommends Viva web-app upload for Viva Insights customers. Use Microsoft Entra plus .csv files for parallel data uploads (visible date 30 June 2025; content refreshed 7 July 2026) says add/edit uploads are no longer supported there, while delete and Replace All remain. Upload and maintain data through the Microsoft 365 admin center (5 December 2025) says migration timing is tenant-specific and communicated through Message Center. Do not silently resolve this conflict.

Surface map — where each URL takes you

Use this to confirm you’re not asking the dashboard view to host the admin UI:

analysis.insights.cloud.microsoft (Copilot Dashboard page) Same destination as the short link above. Adoption / Readiness / Impact pages. No — ever.

30-second diagnosis

  1. Read the latest organizational-data migration post in Message Center; record the post ID and date.
  2. Open admin.microsoft.com → Setup → Migration and imports. Record whether Organizational Data in Microsoft 365 and Get started/New import are present.
  3. Open analysis.insights.cloud.microsoft → Admin → Data hub. Record which add/edit, delete, and Replace All controls are present.
  4. Review current Data Connections so a competing source is not introduced accidentally.
  5. If Message Center and the UI agree, confirm the corresponding role and refresh the sign-in token. If they conflict or neither surface offers add/edit, stop and follow the escalation package.

Causes ranked by likelihood

#CauseWhy it happensFix & verification
1 Looking at the wrong page within the right family of admin surfaces. The Copilot Dashboard view (which is what aka.ms/copilotdashboard redirects to) never shows an “Upload org data” control — for any role. The Copilot Dashboard view is a consumption surface. Organizational-data controls are on the Microsoft 365 admin center Data Connections page or, where the tenant explicitly retains the capability, the Viva Insights admin Data hub.
  • Microsoft 365 admin center — admin.microsoft.com → Setup → Migration and imports → Organizational Data in Microsoft 365 → Data Connections.
  • Viva Insights admin — use analysis.insights.cloud.microsoft → Admin → Data hub only when Message Center and the live UI explicitly support add/edit.
  • Verify: the upload wizard loads with a file picker and an attribute-mapping step.
2 The Microsoft 365 admin-center path is active and the user only holds Insights Administrator. Organizational Data in Microsoft 365 uses a dedicated ingestion role. Insights Administrator governs Viva Insights admin features but does not grant Microsoft 365 admin-center import rights. Microsoft documents Global Administrator and Organizational Data Source Administrator for this surface.
  • Assign Organizational Data Source Administrator in Entra ID: admin.microsoft.com → Users → Active Users → select user → Manage roles → search for “Organizational Data Source Administrator.”
  • Use Organizational Data Source Administrator for ongoing operations. Use Global Administrator only as a temporary diagnostic path, then return to least privilege.
  • Verify: after sign-out/sign-in, the “Organizational Data in Microsoft 365” node appears under Setup → Migration and imports, and the Get started / New import button is enabled.
  • Source: Microsoft Learn — Organizational Data in Microsoft 365: Roles and permissions (and the Organizational Data Source Administrator details section).
3 Role was just assigned — token has not refreshed. Entra role activations propagate to access tokens on next sign-in or token refresh, which can take up to 60 minutes if the user keeps an existing session open.
  • Sign out of all Microsoft 365 surfaces, close the browser, and sign back in (incognito window is fastest).
  • Verify: open myaccount.microsoft.com → My roles — the newly assigned role should be listed before reopening the admin surface.
4 Role is granted via Privileged Identity Management (PIM) but not activated. If the org uses PIM, the user may be eligible for the role but not currently holding it. Eligibility alone does not surface admin UI.
  • portal.azure.com → Privileged Identity Management → My roles → activate the role required by the tenant-enabled surface: Organizational Data Source Administrator for Microsoft 365 admin-center imports, or Insights Administrator for a Viva add/edit path that is explicitly present and supported.
  • Verify: activation appears under Active assignments with a remaining timer.
5 Role is scoped to a custom Administrative Unit (AU) rather than the tenant. An Administrative Unit-scoped role cannot perform tenant-wide ingestion. The Organizational Data in Microsoft 365 Data Connections node is tenant-scoped, so the UI can be hidden for scoped admins.
  • In Entra ID → Roles and administrators, locate the assignment and confirm the Scope column shows “Directory” (tenant), not an AU name.
  • If scoped, re-assign at directory scope (requires a Privileged Role Administrator or Global Administrator).
6 Wrong account in the browser session. Multi-account browsers (work + personal, or two work tenants) often re-use a cached account silently — the dashboard loads under the unprivileged identity.
  • Open aka.ms/copilotdashboard in an incognito / InPrivate window and sign in explicitly with the admin account.
  • Verify: top-right account chip shows the correct UPN.
7 Ingestion is already handled by a connector or Azure Blob source — manual CSV is intentionally hidden / disabled. An existing SAP SuccessFactors, Workday, Azure Blob, API, or CSV connection may already own the target-app attributes. Availability and permitted source combinations are shown in the live Data Connections experience.
  • admin.microsoft.com → Setup → Migration and imports → Organizational Data in Microsoft 365 → Data Connections → review the existing connection list.
  • Do not pause, remove, or add a connection until the source owner, downstream apps, attribute precedence, and rollback have been documented. Update the existing authoritative source when appropriate.
  • See the Upload Methods Overview in the Data Onboarding Guide.
8 Viva Feature Access Management (VFAM) policy disables the Copilot Dashboard surface for this user. VFAM gates the viewing surface itself. The user lands on a permission-denied page or a stripped-down view that does not even expose admin entry points if they were ever there.
  • admin.microsoft.com → Viva → Feature Access Management → Viva Insights → verify the Viva Insights web app control (pre-rollout tenants: Copilot Dashboard) is ON for the user’s group.
  • Remember: this affects the Viva Insights web-app viewing surface, not the Microsoft 365 admin-center ingestion role. A tenant-enabled Viva admin path still depends on its own role and current UI.
9 Tenant under 2,500 licensed users without the required admin role. Small-tenant guardrails: certain admin entry points are gated until the user holds AI Administrator, Insights Administrator, or Global Administrator.
  • Confirm the exact role required by the surface you are testing. Do not treat AI Administrator, Insights Administrator, and Organizational Data Source Administrator as interchangeable.
  • Verify: myaccount.microsoft.com → My roles.
Most common misdiagnosis

Do not diagnose a permissions defect from the Copilot Dashboard page. First separate wrong surface, tenant rollout state, role scope/token, and existing connection ownership. Message Center plus both live admin surfaces are the controlling evidence.

Supported correction and rollback after access is restored

  • Use incremental CSV updates for affected users; omitted or blank values preserve existing data.
  • Use documented type-specific deletion markers for values that must be removed: '' for strings in CSV, -1 for integers, a nonexistent in-tenant address for email, and 01-01-0001 for dates.
  • Use Historical import with File valid as of for retroactive Viva Insights corrections.
  • To return Viva ManagerId or Organization to Entra, enable those attributes under Configure Entra connection, then remove optional uploaded attributes no longer needed.
  • Do not use a one-row dummy Replace All file as a general rollback. For critical migration failures, Microsoft documents temporary rollback only through the support channel in special cases.

Sources: Import and manage your organizational data with a .csv file — Update or make other changes and Make retroactive updates (Microsoft Learn, updated 2 April 2026); Use Microsoft Entra plus .csv files for parallel data uploads — FAQs (visible date 30 June 2025; refreshed 7 July 2026); and Upload and maintain data through the Microsoft 365 admin center — Support and rollback (5 December 2025).

Stop and escalate

Open Microsoft Support or engage the account team when Message Center and the UI conflict, neither surface offers add/edit, the required role is active at directory scope but the expected control remains absent after token refresh, or processing exceeds the documented window. Include:

  • tenant ID and cloud; Message Center post ID/date
  • UTC timestamps, affected UPN/role/PIM scope, and sign-in refresh result
  • screenshots of both admin surfaces and Data Connections
  • connection/import IDs, validation logs, correlation IDs, and expected versus actual behavior

What “done” looks like

  • Message Center and the live tenant UI identify the supported add/edit surface.
  • User signs into that surface and holds its matching directory-scoped role.
  • Sees and clicks Get started / New import / .csv upload → Start, completes the wizard, and the job appears with status Validating → Processing → Complete.
