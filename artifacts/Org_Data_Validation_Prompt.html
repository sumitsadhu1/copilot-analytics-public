Both formats produce the same validation. The modular flow splits the analysis into three sequential steps to fit within M365 Copilot input limits. Always attach or paste the customer’s org data file (Excel or CSV) with each prompt.

1. Modular Flow — 3 Prompts

How the Flow Works

Run these three prompts in order, attaching the same org data file to each. Each prompt produces a self-contained report you can save. The prompts guide you to the next step automatically.

Prompt 1: Data Readiness  ➔  Prompt 2: Dashboard Impact  ➔  Prompt 3: Remediation Plan

Prompt 1 of 3 — Data Readiness Assessment

Purpose: Is the file structurally ready for upload? Checks column names, data types, hierarchy integrity, and data quality.

# ROLE You are a Microsoft Viva Insights Organizational Data Validation Specialist. Analyse the attached org data file and produce a DATA READINESS REPORT. # TASK Validate this org data file for upload readiness to the Copilot Dashboard / Viva Insights platform. # UPLOAD PATH DETECTION - If columns start with "Microsoft_" → M365 Admin Centre (MODIS) path - If columns use PersonId/ManagerId/Organization → Viva Insights (Legacy) path - If NEITHER matches → CRITICAL: non-standard column names. Produce a rename mapping. # REQUIRED ATTRIBUTES (Legacy) | Attribute | Type | Rules | |---|---|---| | PersonId | Email | Must contain @, unique per EffectiveDate | | EffectiveDate | DateTime | MM/DD/YYYY or DD/MM/YYYY | | ManagerId | Email | Must contain @, blank only for CEO row | | Organization | String | Non-blank for every row | # REQUIRED ATTRIBUTES (MODIS) PersonId → Microsoft_PersonEmail, ManagerId → Microsoft_ManagerEmail, Organization → Microsoft_Organization. EffectiveDate is set via UI date picker (no column needed). # RESERVED OPTIONAL ATTRIBUTES If present, must use exact names and correct types: | Attribute | Type | Rules | |---|---|---| | FunctionType | String | Job function category | | LevelDesignation | String | Seniority level | | HireDate | DateTime | No #N/A or sentinel strings | | HourlyRate | Double | Numeric only, no $ or commas. Set to Confidential during upload. | | Layer | Integer | 0,1,2,3... not strings | | SupervisorIndicator | String | IC, Mngr, or Mngr+ only | | Location | String | Office location | | CountryOrRegion | String | Country | | WorkDayStart | Integer | 0-23 only. No decimals, no time strings ("09:00"). Drives after-hours / focus-time metrics. | | WorkDayEnd | Integer | 0-23 only, must be > WorkDayStart per row. | | StartDate | DateTime | Record validity start date (distinct from HireDate). Must be <= EffectiveDate. | | EndDate | DateTime | Record validity end date. Must be >= StartDate. | | TimeZone | String | IANA TZ (e.g. America/New_York). NOT "PST"/"EST". | | IsActive | Boolean | TRUE / FALSE only | | IsInternal | Boolean | TRUE / FALSE only | # FORBIDDEN COLUMN NAMES Truly forbidden (will be rejected or cause upload errors): Domain, PopulationType, ObjectId, MetricDate, StandardTimeZone, WeekendDays, InferredTeamSize, PrimaryCollaborator_PersonId, SecondaryCollaborator_PersonId Note: TimeZone, StartDate, EndDate, IsActive, IsInternal are reserved (not forbidden) — see table above for rules. # VALIDATION CHECKS 1. Required columns exist and complete 2. PersonId valid email format (contains @) 3. ManagerId valid email (blank only for CEO) 4. Duplicate PersonIds 5. Date format detection (MM/DD vs DD/MM), flag ambiguous dates 6. Sentinel values: #N/A, N/A, NULL, "null", "none", 0 as placeholder 7. Placeholder dates: 01/01/1900, 01/01/1970, 12/31/9999 8. Hierarchy integrity: orphan managers, loops 9. Reserved attribute data types 10. Unique values per column — flag >200 or <3 11. Parenthetical IDs in values (e.g. "Sales (10001)") 12. Total columns <= 105 13. Forbidden column names 14. Columns with >50% blank values 15. HRIS indicators: if columns suggest SAP SuccessFactors or Workday, recommend native connector 16. WorkDayStart / WorkDayEnd: integer 0-23 only; WorkDayStart < WorkDayEnd per row; flag if >30% of rows share the same value (suspicious mass-default from HRIS); flag if >5% rows = 0 (likely blank→0 sentinel). These trigger the Data Quality page warning triangle and degrade after-hours / focus-time metrics. 17. StartDate <= EffectiveDate <= EndDate per row (record validity window) 18. PII column-name detection: flag columns matching (ssn|social|tax_id|nino|phone|mobile|dob|birth|home_addr|home_address|emergency|bonus|salary_ytd|compensation) — these must NOT be uploaded to org data # OUTPUT FORMAT — MANDATORY Produce ALL SIX sections below as VISIBLE MARKDOWN TEXT in the response. Do NOT stop after any single table. Do NOT summarise findings into "analysis complete" — every finding from your reasoning MUST appear in the visible output. The completeness table alone is NOT a complete report. ## 1. Upload Readiness Verdict State the verdict FIRST, in the first line of your response: PASS / PASS WITH WARNINGS / FAIL. Follow with one sentence explaining the top reason. ## 2. Critical Issues (Plain Bulleted List — MANDATORY) Before any table, list every CRITICAL finding as plain bullets. Include at minimum (if present in the file): - Forbidden column names detected (list them) - PII columns detected (list them — SSN, DOB, HomeAddress, Bonus, etc.) - Dual-schema detected (both legacy PersonId and Microsoft_PersonEmail present) - Duplicate PersonIds on the same EffectiveDate (list row numbers) - Orphan managers (ManagerId values not present as PersonId — list them) - Manager self-references or hierarchy loops (list them) - Mixed date formats in EffectiveDate (count of distinct formats) - WorkDayStart / WorkDayEnd invalid values (list sample offending values) - Sentinel strings in HireDate (#N/A, NULL, none, TBD, unknown, 01/01/1900) ## 3. File Summary File name, format, total rows, total columns, EffectiveDate range, unique PersonId count, upload path detected (Legacy / MODIS / Ambiguous). ## 4. Attribute Completeness Table: Column Name | Filled Rows | Total Rows | Completeness % | Quality Quality definition: "Good" requires BOTH ≥95% completeness AND no value/type/format violations. A column with 100% fill but invalid values (sentinels, type mismatches, wrong format) is "Low" or "Poor", never "Good". Reserved-attribute type rules (integer, boolean, IANA, email, etc.) must be applied. ## 5. Required & Reserved Attributes — Validation Table: Attribute | Present | Type/Format Check | Sample Values | Issue Found ## 6. Full Data Quality Issue List Numbered list of every issue, severity (CRITICAL / WARNING / INFO), and specific fix instruction. Aim for 15+ items on adversarial files; if fewer, explain why. ## ▶ NEXT STEP End with: "Save this report. Then run Prompt 2: Dashboard Impact Simulation with the same file attached to see how this data will appear on the Copilot Dashboard." # ANTI-TRUNCATION RULES - If you produce Python-based analysis, the rendered table is NOT sufficient. You MUST also produce sections 1, 2, 5 and 6 as visible Markdown text. - Never end the response with "analysis has been fully executed" without the full six sections above. - If the file is too large to analyse exhaustively, state that explicitly in Section 1 and produce what you can for every section. Now analyse the attached file. Begin your response with the verdict on the first line.

2. Full Prompt — Single Prompt (Advanced)

For Large-Context AI Platforms

This is the original monolithic prompt (~18,000 characters). Use with GitHub Copilot (Opus/Claude), ChatGPT, or Claude which can handle the full prompt + file in a single pass. Not recommended for Microsoft 365 Copilot Chat or the Analyst Agent due to input size limits.

Copy the entire prompt below. When using it, attach the customer's org data file (Excel/CSV) along with the prompt.

# ROLE You are a Microsoft Viva Insights Organizational Data Validation Specialist. Your job is to analyse a customer's organizational data file (CSV or Excel) and validate it against Microsoft's official requirements for the Copilot Dashboard and Viva Insights platform. # TASK Analyse the attached organizational data file and produce a comprehensive validation report covering: 1. Required attribute compliance 2. Reserved attribute data type validation 3. Data quality issues 4. Copilot Dashboard filter recommendations 5. Specific fix instructions # UPLOAD PATH DETECTION Before validating, determine which upload path the file is intended for: Viva Insights (legacy path) — uses column names: PersonId, ManagerId, Organization, EffectiveDate, FunctionType M365 Admin Centre (new MODIS path) — uses column names: Microsoft_PersonEmail, Microsoft_ManagerEmail, Microsoft_Organization, Microsoft_JobDiscipline Detection logic: - If columns start with "Microsoft_" → file is for M365 Admin Centre path. Validate against the MODIS schema below. - If columns use PersonId/ManagerId/Organization → file is for Viva Insights legacy path. Validate against the legacy schema below. - If NEITHER pattern matches → the file has non-standard column names. Report as CRITICAL and provide a mapping table for both paths. # SCHEMA — VIVA INSIGHTS (LEGACY PATH) Required columns: PersonId, EffectiveDate, ManagerId, Organization # SCHEMA — M365 ADMIN CENTRE (MODIS PATH) Required columns: Microsoft_PersonEmail, Microsoft_ManagerEmail, Microsoft_Organization Note: EffectiveDate is NOT a column — it is selected via the UI date picker during upload. Note: FunctionType → Microsoft_JobDiscipline, Location → Microsoft_CompanyOfficeLocation, HireDate → Microsoft_HireDate, HourlyRate → Microsoft_HourlyRate, LevelDesignation → Microsoft_LevelDesignation Note: No spaces allowed in column headers (use underscores). Each must be present as a column header (exact spelling, case-sensitive) and every row must have a non-blank value: | Attribute | Data Type | Description | |----------------|-----------|-------------| | PersonId | Email | Unique identifier — must be primary SMTP address or email alias (e.g., joe@contoso.com) | | EffectiveDate | DateTime | Date the attribute values apply. Supports: MM/DD/YYYY, DD/MM/YYYY, YYYY/MM/DD. If missing, upload date becomes default. | | ManagerId | Email | Manager's primary SMTP address. Can be blank for CEO only. | | Organization | String | Internal organization/business unit. MUST have non-blank values for every row. | # MICROSOFT LEARN REFERENCE — RESERVED OPTIONAL ATTRIBUTES These are RESERVED OPTIONAL. If included, they must use these exact column names and correct data types: | Attribute | Data Type | Description | Dashboard Impact | |------------------------|-----------|-------------|-----------------| | FunctionType | String | Job function (e.g., Finance, Engineering, Sales) | Enables "Job Function" filter on Copilot Dashboard | | LevelDesignation | String | Seniority level (e.g., Director, VP, IC) | Used in some Power BI templates | | HireDate | DateTime | Employment start date. #N/A or non-date values will fail. | Used in tenure-based analysis | | HourlyRate | Double | Salary as hourly rate in USD. No commas or $ signs. | Used in Copilot Assisted Value calculations. ACCESS: Set to Confidential during upload so only authorised analysts can see salary data — not all dashboard viewers. | | Layer | Integer | Hierarchy distance from CEO (0 = CEO, 1 = direct reports, etc.) | Must be a NUMBER, not a string | | SupervisorIndicator | String | "IC", "Mngr", or "Mngr+" | Used in manager/IC analysis | | WeeklyBadgeOnsiteDays | Double | 0–7 or -1 if unknown | Used in hybrid work analysis | | Location | String | Office location | Used in location-based filtering | | CountryOrRegion | String | Country or region where the employee works | Used in regional analysis and benchmarks | | WorkDayStart | Integer | 0-23 only. No decimals, no time strings. Flag if >30% rows share same value (mass-default from HRIS) or >5% rows = 0 (blank→0 sentinel). | Drives after-hours, focus-time, collaboration-hours metrics. Invalid values trigger Viva Insights Data Quality warning triangle within 48h post-upload. | | WorkDayEnd | Integer | 0-23 only, must be > WorkDayStart per row. | Pairs with WorkDayStart. Same post-upload quality-score impact. | | StartDate | DateTime | Record validity start date (distinct from HireDate). StartDate <= EffectiveDate. | Used in time-validity filtering. | | EndDate | DateTime | Record validity end date. EndDate >= StartDate. | Used in time-validity filtering. | | TimeZone | String | IANA TZ format (e.g. America/New_York). NOT "PST"/"EST". | Used in after-hours calculations. | | IsActive | Boolean | TRUE or FALSE only. | Used to filter active vs terminated employees. | | IsInternal | Boolean | TRUE or FALSE only. | Used to separate employees from external collaborators. | # COPILOT DASHBOARD FILTER MAPPING The Copilot Dashboard has three main filters. Map customer data to these: | Dashboard Filter | Source Attribute | Entra ID Fallback | |-----------------|-----------------|-------------------| | Scope (Your company / Your group) | ManagerId → builds hierarchy tree | Entra ID "ManagerID" attribute | | Organization filter | Organization column in CSV | Entra ID "Department" field | | Job Function filter | FunctionType column in CSV | Does NOT fall back to Entra — only appears if FunctionType is uploaded | # VALIDATION RULES Apply these checks to the file: 1. RESERVED ATTRIBUTE DATA TYPE CHECKS - Layer: Must be Integer (0, 1, 2, 3...). Check for $, commas, currency symbols → ERROR: remove formatting - SupervisorIndicator: Must be "IC", "Mngr", or "Mngr+" → WARNING if other values used 3. Flag the format detected and advise that the admin must select the matching DateTime format during upload (DateTime_MMDDYYYY or DateTime_DDMMYYYY). Ambiguous dates (e.g., 03/04/2026 could be March 4 or April 3) should be flagged. - Check for suspicious placeholder dates: 01/01/1900, 01/01/1970, 12/31/9999, 0/01/1900 — flag as WARNING even if parseable, since they indicate missing real data. - Check EffectiveDate: all same date? Multiple snapshots? Recommend monthly snapshots for historical analysis - Check for blank/missing ManagerId chains that would break the hierarchy (Scope/Group filter) - Check for manager hierarchy loops (A reports to B and B reports to A) — these cause upload errors. If hierarchy cannot build (no managers exist as PersonIds), note that loop detection is not applicable and explain why. - Flag columns with >50% blank values as low quality - Confirm no truly forbidden system field names are used as column headers: Domain, PopulationType, ObjectId, MetricDate, StandardTimeZone, WeekendDays, InferredTeamSize, PrimaryCollaborator_PersonId, SecondaryCollaborator_PersonId, or any name starting with "Microsoft_" (unless mapping to a known reserved attribute). - Note: TimeZone, StartDate, EndDate, IsActive, IsInternal, WorkDayStart, WorkDayEnd are RESERVED (not forbidden) — validate against the reserved-attribute rules above, do not flag as forbidden. - WorkDayStart / WorkDayEnd specific checks: integer 0-23 only; WorkDayStart < WorkDayEnd per row; flag if >30% of rows share the same value (suspicious mass-default from HRIS); flag if >5% of rows are 0 (likely blank→0 sentinel). These trigger the Viva Insights Data Quality page warning triangle within 24-48h post-upload and degrade after-hours / focus-time / collaboration-hours metrics. - StartDate / EndDate specific checks: StartDate <= EffectiveDate <= EndDate per row (record validity window). - TimeZone specific check: must be IANA format (e.g. America/New_York, Europe/London), not short codes like PST/EST/GMT. - PII column-name detection: flag columns matching patterns (ssn|social|tax_id|nino|phone|mobile|dob|birth|home_addr|home_address|emergency_contact|bonus|salary_ytd|compensation) → CRITICAL: these must NOT be uploaded to org data. Remove before upload. - Report total attribute count vs the 105 maximum 4. DASHBOARD FILTER RECOMMENDATIONS Based on the columns and values in the file, recommend: - Which column should map to Organization (required) — prefer the top-level business unit/division column - Which column (if any) should map to FunctionType — prefer mid-level functional area - Which column (if any) should map to LevelDesignation — prefer seniority/grade - Which columns should remain as custom attributes for analyst queries - Whether the customer has enough hierarchy depth for meaningful Scope/Group filtering 5. CUSTOM ATTRIBUTE REVIEW - Identify all non-standard columns (not in required or reserved lists) - Verify custom attribute names don't conflict with reserved names - Total attribute count must not exceed 105 (including 5 required + reserved) - Flag any custom attributes containing PII (SSN, date of birth, home address) — these should NOT be uploaded 6. HRIS CONNECTOR RECOMMENDATION Check whether the column names or values suggest the data originates from a known HRIS platform. If detected, recommend the native M365 connector instead of CSV upload — this eliminates the entire manual validation and upload workflow. SAP SuccessFactors indicators: columns named userId, jobCode, businessUnit, division, department, costCenter, employmentNav, or values containing "SF-" prefixes, SuccessFactors-style employee IDs. → Recommend: "Your data appears to originate from SAP SuccessFactors. Microsoft 365 provides a native SAP SuccessFactors connector that syncs org data automatically on a weekly/monthly schedule — no CSV preparation or upload needed. See: https://learn.microsoft.com/viva/organizational-data/sap-successfactors-connector" Workday indicators: columns named Worker_ID, Supervisory_Organization, Cost_Center_Reference, Management_Level, or values containing Workday-style WD/WID identifiers. → Recommend: "Your data appears to originate from Workday. Microsoft 365 provides a native Workday connector (for tenants with ≤100K users) that syncs org data automatically. For larger tenants, use the Workday RaaS → Azure Blob Storage path. See: https://learn.microsoft.com/viva/organizational-data/workday-connector" If no HRIS platform is detected, skip this section silently — do not mention it. # OUTPUT FORMAT Produce the report in this structure: ## 1. File Summary - Total rows, total columns, file format - Date range (EffectiveDate values) - Estimated employee count (unique PersonIds) - Upload path detected: Viva Insights (legacy) or M365 Admin Centre (MODIS) - If MODIS path detected with legacy column names, include the Column Rename Mapping table ## 1b. Attribute Completeness For EVERY column in the file (not just required/reserved), produce a completeness table: | Column Name | Filled Rows | Total Rows | Completeness % | Quality | Where Quality = "Good" (>95%), "Low" (50-95%), "Poor" (<50%), or "Empty" (0%). This enables ongoing data quality audits — the customer can track coverage trends month over month. ## 2. Required Attributes — Pass/Fail Table showing each required attribute, whether it exists, completeness %, and any issues. ## 3. Reserved Attributes — Validation Table showing each reserved attribute found, data type check (pass/fail), sample values, and fix needed. ## 4. Data Quality Issues Numbered list of every issue found, severity (CRITICAL / WARNING / INFO), and specific fix instruction. ## 5. Dashboard Filter Mapping Recommendation Table showing: Dashboard Filter → Recommended Source Column → Sample Values → Action Required. Include rationale for each recommendation. ## 6. Column Mapping Summary Complete table of ALL columns in the file: | Customer Column | Map To | Action | Rationale | Show what to keep, rename, populate, or remove. ## 7. Fix Checklist Numbered, actionable checklist the customer or analyst can follow to fix the file before upload. Order by priority (critical fixes first). ## 8. Dashboard Impact Assessment For EACH data quality issue found, describe the specific impact on the Copilot Dashboard experience using this table format: | Data Issue | Dashboard Page Affected | What the User Will See | Severity | |---|---|---|---| Use these rules to determine dashboard impact: - Organization empty/missing → Adoption + Impact pages: Organization filter will not appear OR will show only Entra Department values. "View by: Organization" will be empty. Leaders cannot segment adoption by business unit. - FunctionType empty/missing → Adoption + Impact pages: Job Function filter will NOT appear at all (no Entra fallback). "View by: Job Function" unavailable. Cannot compare adoption across job roles. - ManagerId hierarchy broken (orphaned managers) → All pages: Scope filter will show only "Your company" with no group breakdowns. "Your group" view will show 0 people for most leaders. Senior leader list will be empty or show only 1 person. - Layer invalid → Upload fails if partial ingestion is off. If renamed to custom attribute, no direct dashboard impact but Power BI templates expecting Layer (integer) will break. - HireDate invalid → Upload may fail or rows rejected. Tenure-based segmentation in Power BI templates will be incomplete. No direct dashboard page impact. - HourlyRate missing → Impact page: Copilot Assisted Value will use the $72/hr session-only default (per-viewer session setting, not a persistent tenant setting) instead of organisation-specific rates. ROI calculations may be inaccurate across viewers. - HourlyRate present but not Confidential → Security risk: If uploaded with Public access, salary data will be visible to all dashboard viewers and selected apps. Always recommend setting HourlyRate to Confidential access during upload (M365 Admin Centre path) to restrict visibility to authorised Viva Insights analysts only. - Organization values are system codes (CC-4023, DEPT001) → Adoption + Impact pages: Filter dropdowns and "View by: Organization" labels will show cryptic codes instead of readable business unit names. Leaders won't recognise the values. Recommend mapping codes to human-readable names before upload. - Single EffectiveDate (no monthly snapshots) → Adoption + Impact pages: 6-month trendlines on the dashboard are unaffected (they use M365 telemetry data, not org data). However, Advanced Analytics (Power BI person queries) will only have one point-in-time org data snapshot, limiting historical cohort analysis. - Parenthetical IDs in values → Adoption + Impact pages: Filter dropdowns and "View by" labels will display cluttered text like "Corporate Services (10000593)" instead of clean "Corporate Services". - Too many unique values in Organisation (>50) → Adoption page: Filter dropdown becomes unusually long and hard to navigate. Consider using a higher-level grouping. - Too few unique values in Organisation (<3) → Adoption page: Limited analytical value — cannot meaningfully compare across groups. - >50% blank values in a column → Quality score: Data quality score in Viva Insights will show below 95% for that attribute, and missing employees are excluded from filtered views using that attribute. - LevelDesignation empty → No dashboard impact (not used as a dashboard filter). Impacts Power BI templates that segment by seniority level. - Location/CountryOrRegion empty → No dashboard filter impact unless used in custom reports. Geographic analysis in Power BI will be unavailable. - Minimum group size suppression → All pages: Groups with fewer than 10 employees (default) will show "--" instead of metrics. This is privacy protection, not a data error. If many groups show "--", the org data may be too granular for the dashboard. - WorkDayStart / WorkDayEnd invalid, out-of-range, low variance, or >5% zero-sentinels → Data Quality page (Analysis setup): warning triangle appears next to the attribute within 24-48h; quality score drops below 95%. Advanced Insights: after-hours, focus-time, and collaboration-hours calculations become inaccurate for affected employees. - StartDate or EndDate outside validity window (EffectiveDate not between StartDate and EndDate) → Data Quality page: rows may be excluded from analytics. Warning triangle on affected attribute. - TimeZone in short format (PST/EST/GMT) instead of IANA → Data Quality page: warning triangle; after-hours calculations may fall back to tenant default timezone. - PII columns present (SSN, home_address, bonus, DOB) → CRITICAL before upload: remove columns. These must never be uploaded to org data regardless of dashboard impact — it's a data-governance breach. Organization column empty Upload fails validation or Organization filter missing Populate from Groups/Department column or remove column to fall back to Entra
Layer has string values Upload rejects Layer column or data type error Rename column to custom attribute (e.g., “PayBand”) and create a numeric Layer if needed

Scope shows only one group “View by: Group” shows one leader only Ensure ManagerId chain is complete — every manager must also be a row in the file

Duplicate PersonIds Upload may fail or data overwritten Each PersonId should appear once per EffectiveDate. Remove duplicates.
Custom attribute name conflicts Upload fails with naming error Custom attributes cannot reuse reserved names (Organization, Layer, FunctionType, etc.)
