Copilot Analytics with Agency-Default Views for Multi-Agency Single Tenant

Version: 1.1 | Author: Sumit Sadhu | Last validated: 16 July 2026

THIS GUIDE’S AUTHOR-CREATED ARCHITECTURE PATTERN
Default Priority is an editorial name created by this guide. It is not a Microsoft product, security, compliance, sovereignty, or licensing term. The companion Restricted Reporting pattern was formerly labelled “Isolation”; the old word remains in the existing filename and URL only for link stability. Microsoft documents the component capabilities, but does not document either combined pattern as an end-to-end security boundary.

This pattern intentionally accepts that some authorized users can obtain broader cross-agency or tenant-wide views. Manager hierarchy can establish a useful default scope, but it is not an authorization boundary. Viva Insights partitions limit rows and columns in an assigned analyst query workspace; separate controls govern Dashboard, Analytics Reports, Agent Dashboard, metrics export, delegation, manager access, privileged roles, downloaded results, and Power BI. Neither pattern creates cryptographic, tenant, product-security, or independently certified isolation.

Supported-environment and licensing gate:
Supported Microsoft 365 environments for Viva Insights features (30 June 2026) says GCC does not support Agent Dashboard, Copilot metrics export, or Copilot Power BI reports. Connect to the Agent Dashboard (preview) (16 June 2026) says Agent Dashboard is unavailable in national and regional clouds. Plan-level support does not establish feature parity. Verify the exact cloud, tenant UI, Message Center, assigned service plans, agent activity, and every required surface before design.

Microsoft pages describe the advanced-access threshold differently. Environment requirements for Viva Insights (12 May 2026) explicitly gives 30 Microsoft 365 Copilot + 20 Viva Insights as 50 assigned Viva Insights service plans; the Dashboard article (1 July 2026) describes 50 Copilot or 50 Viva Insights. Do not state that mixed assignments fail. Verify assigned plans and tenant availability. For Agent Dashboard, use the conservative feature-specific prerequisite of 50 Microsoft 365 Copilot licences plus agent activity; a 50-Viva-Insights-only tenant must validate availability. Escalate a documented mismatch to Microsoft Support.
Organizational-data precedence gate:
An upload does not universally or permanently replace Entra. Uploaded values change the source for supplied dashboard attributes; parallel ingestion can keep or restore Entra for ManagerId and Organization; Microsoft 365 Profile precedence is separate. Before change, capture each required attribute, current source, owner, effective date, coverage, and target surface in Data Quality and Profile. Check Setup → Migration and imports, Message Center, and the actual tenant UI. Use Organizational Data in Microsoft 365 for new/edited uploads unless the tenant explicitly presents a supported legacy path. Use supported update/delete behavior, not dummy rows. Source basis: Use your organizational data in Microsoft 365 and Microsoft Viva (2 December 2025), Upload and maintain data through the Microsoft 365 admin center (5 December 2025), Use Microsoft Entra plus .csv files for parallel data uploads (30 June 2025), Import and manage your organizational data with a .csv file (2 April 2026), Import organizational data from Workday (6 April 2026; 100,000-user limit), and Import organizational data using API-based import (preview) (6 April 2026).

Mandatory Negative Tests, Evidence, and Release Veto

  1. Test agency leader, cross-agency manager, shared executive, delegate, agency analyst, multi-partition analyst, global-partition analyst, Power BI Viewer and Member/Contributor, AI Administrator, Insights Administrator, Insights Analyst, and Global Administrator accounts.
  2. Test Copilot Dashboard, Analytics Reports, Agent Dashboard, weekly/daily metrics export, advanced query workspace, query-result download, delegation, manager-published reports, Power BI workspace/app, report export, Analyze in Excel, subscriptions, and resharing.
  3. Capture test date, tester identity/role, group and partition memberships, policy/access export, org-data snapshot, expected result, observed result, screenshot or audit record, Power BI permissions, owner, and remediation ticket.
  4. Any failed or unexecutable negative test vetoes publication and go-live. Restore the pre-change state or remove access, preserve evidence, and open Microsoft Support when documented and observed behavior differ.
  5. Obtain written Security, Compliance/Legal, Privacy/DPO, Viva Insights owner, and Power BI owner sign-off on accepted visibility and residual risk. Repeat the suite quarterly and after any hierarchy, role, delegation, org-data, partition, VFAM, licence, cloud, preview-status, or sharing change.

Rollback/change control: before change, export or capture VFAM policies, access lists, role assignments, delegations, hierarchy, partition filters/analysts, organizational-data source/quality, query assets, and Power BI permissions. Define owner, maintenance window, approvers, success/failure criteria, and restoration order. Enabling partitions is a high-impact tenant change; turning partitions off requires Microsoft Support. Export query assets before enabling and confirm deletion impact before removing any partition.

TL;DR — Agency-Default Views in Under a Day

Leverage native Copilot Dashboard scoping and delegation so each agency leader sees their own data by default — no lockdowns, no partitions required.

1. Executive Summary

This document provides an author-created reporting pattern for a single Microsoft 365 tenant that hosts multiple agencies, where each agency wants to see its own Copilot adoption and impact data by default and explicitly accepts broader authorized cross-agency or tenant-wide visibility.

Key Difference from Restricted Reporting:
Restricted Reporting applies group-scoped restrictions and partitioned analyst workspaces to reduce cross-agency reporting paths. This guide keeps eligible native features available because broader authorized visibility is accepted. Hierarchy provides a default view, not a security boundary.
Recommended Approach: Native Scoping + Optional Partitions
  1. Copilot Dashboard with “Your group” default — agency leaders auto-identified; see their own hierarchy’s data by default.
  2. Dashboard Delegation — CEOs delegate down to VPs and directors for cascading agency-scoped views.
  3. Publish Reports — filtered by manager hierarchy for automatic agency-scoped delivery.
  4. Partitions (optional) — for analyst-level deep dives scoped to a single agency; analysts may access multiple partitions if collaboration requires it.

How “Your Group” Scoping Works

The Copilot Dashboard identifies senior leaders using the Entra ID manager hierarchy (Auto CxO Identification). Automatic identification applies only to tenants with at least 2,500 Entra ID users and a majority single reporting line; within that, a separate agency’s top leader is auto-identified only if that agency tree is at least 4% of the total tenant. Smaller tenants or sub-4% agencies must grant the CEO access manually — the “Your group” scope still works once access is granted. Leaders with access see two scopes:

ScopeWhat It ShowsDefault?
“Your group”Copilot data for all employees in the leader’s management chainYes — lands on this scope by default when opening the dashboard
“Your company”Tenant-wide Copilot data across all agenciesNo — available via toggle, but not the default view

When each agency CEO is the top of their hierarchy, “Your group” = their entire agency. The dashboard opens to this view by default, immediately satisfying the “see my agency data first” requirement without any lockdown.

What about “Your company”?
Agency leaders with a tenant-wide access source may be able to use “Your company” and see aggregate data across agencies. This pattern explicitly accepts that possibility (FR-06). If it is not acceptable, evaluate Restricted Reporting and its evidence gate.

5. Implementation Runbook

0 Phase 0: Pre-Flight Verification

Audit the current state before configuring anything.

  • Verify manager hierarchy: Confirm each agency CEO has no cross-agency manager relationship. Use Entra portal → Users → select each CEO → check Manager field. Run the PowerShell audit in Appendix A for bulk verification.
  • Confirm assigned service plans: Apply the conflict-safe licensing gate in Pattern Boundary & Release Gates. Verify feature presence after processing; do not infer failure from a mixed licence assignment.
  • Confirm supported environment: Record the exact cloud and verify Dashboard, Agent, export, Power BI templates, and organizational-data path in the tenant UI and Message Center.
  • Identify agency leaders: List each agency’s CEO and their direct reports (VPs). These will be the primary dashboard consumers and delegation targets.
  • Review current VFAM state: Admin Center → Settings → Viva → Viva Insights → Turn Advanced Insights on or off. Document which features are currently enabled or disabled.
  • Check existing dashboard access: Admin Center → Settings → Viva → Viva Insights → Manage user access for Copilot Dashboard. Note who currently has access.
  • Capture rollback baseline: Export policies, access, delegations, role/partition assignments, org-data source/quality, query assets, and Power BI permissions before change.

1 Phase 1: Enable Native Features

Enable only the eligible Dashboard features approved for this accepted-visibility posture. Agent Dashboard and metrics export require separate preview, cloud, privacy, and access decisions.

1.1 — Enable Auto CxO Identification

Admin Center → Settings → Viva → Viva Insights → Turn Advanced Insights on or off → Auto CxO Identification → ensure set to On for Everyone (or verify no disable policy exists).

This automatically identifies agency CEOs (and other senior leaders) based on the Entra ID hierarchy and grants them Copilot Dashboard access with the “Your group” scope.

1.2 — Enable Dashboard Delegation

Same page → Copilot Dashboard Delegation → ensure set to On for Everyone.

This allows agency CEOs to delegate their dashboard view to their direct reports (VPs, directors). Delegates see the same scope as their delegator — the agency’s data.

1.3 — Enable Agent Dashboard

Only if the environment and tenant pass the gate: Agent Dashboard is preview. Treat 50 Microsoft 365 Copilot licences plus agent activity as the conservative prerequisite, validate availability in Worldwide, and do not expect it in national or regional clouds. Scope access to the approved population; do not default to Everyone without a documented cross-agency agent-data decision.

1.4 — Metrics Export (optional)

Copilot metrics export is preview and is not partition-scoped. It requires separate eligibility/global-access validation and is unavailable in GCC. Enable only for an approved group after privacy review and negative testing of weekly/daily exports; otherwise disable CopilotMetricsReport and use governed reports.

After Phase 1 — Verification:
  1. Wait 1–2 hours for AutoCxO to process.
  2. Check Manage user access for Copilot Dashboard — agency CEOs should appear in the list as auto-identified.
  3. Have an agency CEO navigate to insights.cloud.microsoft and confirm they see the Copilot Dashboard with “Your group” as the default scope.
  4. Confirm the “Your group” employee count is in the expected range — note it reflects Copilot-licensed employees only, and groups below the minimum group size (default 10) show “--”, so it tracks licensed headcount, not raw agency headcount.

2 Phase 2: Configure Delegation & Manager Settings

Set up the delegation chain so agency leaders can share their dashboard view with their teams.

2.1 — Agency CEO Delegates to VPs

Each agency CEO should:

  1. Navigate to insights.cloud.microsoft → Copilot Dashboard.
  2. Select the Share or Delegate option (top-right corner of the dashboard).
  3. Add their direct reports (VPs, directors) as delegates.
  4. Delegates will receive access and see the same agency-scoped view (“Your group” inheriting the CEO’s scope).

2.2 — Enable Manager Settings for Publish Reports

Navigate to insights.cloud.microsoft → Admin → Manager settings.

  • Enable managers to receive published reports.
  • Set the minimum team size threshold (this is the same tenant-wide privacy setting).
  • Agency CEOs with team sizes above the threshold will be eligible to receive published reports filtered to their hierarchy.

2.3 — Enrich Organizational Data (Optional)

  • Navigate to Viva Insights → Organizational data → Configure Entra connection → enable ManagerId and Organization checkboxes → Save.
  • If you need custom attributes, first record current source precedence and confirm the tenant-supported upload path; then prepare the data per Appendix B. Uploaded attributes enrich filters but do not govern all access surfaces.
After Phase 2 — Verification:
  1. Have a VP who was delegated access navigate to the dashboard. Confirm they see their agency’s data.
  2. If org data was uploaded, verify filters (Department, Location, etc.) appear on the dashboard.

3 Phase 3: Publish Reports

Use Viva Insights Publish Reports to deliver curated analytics to agency leaders automatically.

3.1 — Run Copilot PBI Template Queries

  • Insights analysts (Central IT or agency-level) navigate to insights.cloud.microsoft → Create analysis.
  • Select a Copilot Power BI template (Copilot Adoption, Impact Explorer, Business Impact, etc.).
  • Run the query. If partitions are enabled, run within the appropriate partition. If not, run against the global population.

3.2 — Publish with “Filter by Manager Hierarchy”

  • After the query completes, select Publish report.
  • Choose “Filter report by manager hierarchy”.
  • Each enabled manager (CEO, VP) will receive a version of the report filtered to their team.
  • Set an expiry date (up to 365 days) and republish monthly to keep data current.
Why this is eligible in Default Priority:
This pattern accepts tenant-wide visibility from documented access sources. Manager publishing is still a separate surface: pilot a shared executive and cross-agency manager, verify Dashboard and report scope, capture evidence, and roll back manager enablement if observed access exceeds the approved posture.

4 Phase 4: Partitions for Analyst Deep Dives (Optional — Mandatory for CIO Reporting)

If your organisation has designated analysts per agency who need to run custom queries, partitions provide row/column-scoped analyst workspaces. This phase is optional for agency-level reporting (dashboard views and published reports may suffice), but needed for the cross-agency query approach described for CIO reporting. A global partition is tenant-wide and requires explicit conflict-of-interest and least-privilege approval. See Section 6.2.

Before Enabling Partitions:
(a) Understand that enabling partitions is irreversible without contacting Microsoft support and will delete all partition configurations if disabled.
(b) Export all existing query results before enabling partitions.
(c) Partitions are needed when analysts require row/column-scoped custom query workspaces. Dashboard and manager-published views may follow hierarchy, but their actual scope must be tested separately.
  • Navigate to insights.cloud.microsoft → Admin → Privacy settings → Partitions → Enable Partitions.
  • Create a partition per agency (e.g., “Agency A”), filtered by Agency attribute (CSV) or Domain (Entra).
  • Assign agency analysts to their primary partition.
  • Cross-agency access: Analysts may be assigned to multiple partitions only with a documented business purpose, conflict-of-interest review, expiry, and quarterly recertification.
  • Assign Central IT analysts to the Global Partition for tenant-wide queries.
After Phase 4 — Verification:
  1. Have an agency analyst sign in and confirm they see their agency’s partition in the partition switcher.
  2. Run a test person query and verify the employee count matches the expected agency headcount.
  3. If a cross-agency analyst exists, confirm they can switch between partitions they are assigned to.

6.1 ROI Analysis per Agency: Capabilities & Gaps

Agency leaders asking “What is Copilot’s ROI for my agency?” will find that some metrics scope naturally to their hierarchy and some do not. Set expectations upfront using the matrix below.

Which ROI Metrics Are Agency-Scoped?

ROI MetricScoped to “Your group” (Agency)?Notes
Copilot Assisted Hours ✅ Yes Impact page shows Assisted Hours for the leader’s management hierarchy. Agency CEOs see their agency’s total.
Copilot Assisted Value ✅ Yes Calculated as Assisted Hours × Hourly Rate. Scoped to “Your group.” However, the hourly rate is a single tenant-wide setting (see limitation below).
Adoption metrics (active users, usage intensity, consistent usage) ✅ Yes Adoption page filters to “Your group” by default.
Sentiment (Glint / Pulse / CSV survey) ❌ No — tenant-level only Sentiment page on the Copilot Dashboard shows tenant-wide data only. Not filterable by “Your group.” Agency leaders see all agencies’ survey responses.
Agent usage (Agent Dashboard) ❌ No — “Your company” only Agent Dashboard has no “Your group” scope. Agency leaders see tenant-wide agent adoption.
Licence cost data ❌ Not in dashboard Licence costs are in M365 Admin Center / billing. Not connected to the Copilot Dashboard. See the Lifecycle & Billing Playbook for cost-side data.
Limitation: Single Tenant-Wide Hourly Rate
Copilot Assisted Value is calculated as Assisted Hours × Hourly Rate. The hourly rate defaults to $72/hour and can be changed only in-session via the dashboard’s Hours and value calculator; the change is not saved and resets to the default each time the dashboard is reopened. It applies to all employees across all agencies — there is no way to persist or set different rates per agency, department, or role.

Impact: If Agency A’s average loaded labour rate is $140/hr and Agency B’s is $65/hr, the Assisted Value figure shown to both CEOs uses the same rate — making one agency’s ROI appear over- or under-stated.

Workaround: Use Power BI templates in partitions (Phase 4) to apply agency-specific hourly rates in the Power BI calculation layer. The raw Copilot Assisted Hours data flows into the PBI template; the analyst replaces the default rate with the agency’s actual loaded labour rate in a DAX measure or parameter. This produces accurate per-agency Assisted Value figures outside the dashboard.

Limitation: No Historical ROI Trending on the Dashboard
The Copilot Dashboard uses a 28-day rolling window and provides no before/after comparison or historical trendline. Agency leaders who need to demonstrate ROI growth over quarters (e.g., “Agency B’s Copilot value grew from $20K/month in Q1 to $35K/month in Q3”) cannot do this from the dashboard alone.

Workaround — Recommended for ROI-focused agencies: Use Power BI templates within partitions (Phase 4). Run the Copilot Impact Explorer or Business Impact template monthly, each time capturing the month’s metrics. Publish each refresh as a new dataset snapshot in the agency’s Power BI workspace. Stack monthly snapshots in a summary report to build a quarter-over-quarter ROI trendline.

Implication: For agencies that need ROI trending, Phase 4 (partitions) is not optional — it becomes a required component of the architecture. Plan the partition and analyst setup accordingly.

Recommendation for ROI-Focused Multi-Agency Deployments
  1. Dashboard for real-time adoption & current-month ROI: Agency CEOs use the Copilot Dashboard (“Your group”) for live adoption metrics and this month’s Assisted Hours/Value.
  2. Partitions + PBI for accurate, historical, agency-calibrated ROI: Treat Phase 4 as mandatory. Analysts run monthly PBI template queries within their agency partition, apply agency-specific hourly rates, and build cumulative ROI reports.
  3. Billing Playbook for cost-side data: Combine the Assisted Value from PBI reports with licence cost data from the Lifecycle & Billing Playbook to produce a net ROI figure per agency.

10. Discovery Questions for Customer

  1. Which cross-agency and tenant-wide views are explicitly acceptable, by actor and surface? If the requirement is tenant, cryptographic, sovereign, or certified security isolation, stop and obtain a security architecture assessment; Restricted Reporting is only defense-in-depth reporting segmentation.
  2. How is the Entra ID manager hierarchy structured? Does each agency CEO sit at the top of an independent tree, or is there a shared parent?
  3. How many agencies, and how many employees per agency? (Each agency needs enough employees to meet the minimum group size threshold for “Your group” to display data.)
  4. Do agency CEOs want to delegate dashboard access to their VPs and directors? (Determines whether to actively configure delegation in Phase 2.)
  5. Are there shared-services teams? Where should they appear — in an agency’s “Your group,” or only in the “Your company” view?
  6. Do you need custom analytical filters (Department, Location, Job Function)? (Determines whether to enrich with CSV org data.)
  7. Will dedicated analysts per agency run custom queries? (Determines whether to enable partitions in Phase 4.)
  8. Are any shared executives, cross-agency managers, global-partition analysts, or analyst conflicts of interest present? Record and test their expected access explicitly.
  9. What is the minimum privacy threshold (minimum group size) acceptable to all agencies? (Tenant-wide setting — all agencies must agree.)
  10. Are Copilot agents deployed across agencies? Agent Dashboard is preview, not partition-scoped, and unavailable in national/regional clouds. Confirm eligibility, approved audience, privacy posture, and tested scope before enabling it.
  11. What exact cloud and preview capabilities are supported? Validate Agent Dashboard, metrics export, Copilot Power BI reports, and organizational-data paths against the current matrix and tenant UI.

# Requires Microsoft Graph PowerShell SDK
Install-Module Microsoft.Graph -Scope CurrentUser
Connect-MgGraph -Scopes "User.Read.All"

# Get all users and their managers — export to CSV for review
Get-MgUser -All -Property Id,DisplayName,UserPrincipalName,Department,Manager `
  -ExpandProperty Manager |
  Select-Object DisplayName, UserPrincipalName, Department,
    @{N='ManagerUPN';E={$_.Manager.AdditionalProperties.userPrincipalName}},
    @{N='ManagerName';E={$_.Manager.AdditionalProperties.displayName}} |
  Export-Csv -Path "HierarchyAudit.csv" -NoTypeInformation

# Find users with no manager set (potential tree breaks)
Get-MgUser -All -Property DisplayName,UserPrincipalName,Manager `
  -ExpandProperty Manager |
  Where-Object { $null -eq $_.Manager } |
  Select-Object DisplayName, UserPrincipalName |
  Format-Table -AutoSize
