Copilot Dashboard & Analytics for Multi-Agency Single Tenant

Version: 2.1 | Author: Sumit Sadhu | Published: March 2026

TL;DR — Two Paths to Agency Isolation

Both paths share the same Phase 1 lockdown. Choose your data strategy based on your environment:

Path A: CSV Org Data UploadPath B: Entra ID Domain Filter

Common Steps (both paths):
  1. Lock down the Copilot Dashboard to Central IT only (VFAM + Entra group) — Phase 1
  2. Choose your data strategy — CSV upload (Path A) or Entra Domain (Path B) — Section 4.1
  3. Create one partition per agency in Viva Insights — Phase 3
  4. Assign agency-specific analysts to their partitions only — Section 7
  5. Deliver reports via Publish Reports or isolated Power BI workspaces — Phase 4

Time estimate: 1–3 days depending on path chosen. Prerequisites: 50+ Copilot licences, Insights Admin role.

1. Executive Summary

This document provides a validated, end-to-end solution architecture for deploying Microsoft Copilot Dashboard and Copilot Analytics in a single Microsoft 365 tenant that hosts multiple independent agencies. Each agency operates as a distinct organization with its own CEO and full management hierarchy, but all share one Entra ID tenant.

Core Constraint & Key Finding:
The customer requires that Agency A cannot see Agency B's Copilot usage data. After validation against official Microsoft documentation, the Copilot Dashboard does not natively support hard data isolation between organizational units within a single tenant. Users identified as "senior leaders" by Entra ID hierarchy will always see "Your company" data, which includes all agencies.
Recommended Approach: A Two-Tier Architecture
  1. Viva Insights Advanced Analytics with Partitions — for agency-isolated Copilot reporting (primary reporting surface for agencies).
  2. Copilot Dashboard with controlled access — for tenant-level oversight by central IT only (secondary, restricted use).

4.1 Choose Your Path: CSV Org Data vs. Entra ID Domain

Viva Insights partitions filter on organizational data attributes. Attributes reach Viva Insights from two sources:

SourceAttributes AvailableSetup Required
Microsoft Entra ID (automatic sync) PersonId, ManagerId, Organization (from Entra Department field), plus system-generated: Domain (from SMTP), TimeZone, PopulationType None — syncs automatically. Optionally enable ManagerId/Organization via “Configure Entra connection”
CSV Org Data Upload All of the above, plus up to 105 custom attributes: Agency, FunctionType, Location, LevelDesignation, CostCentre, etc. Prepare and upload CSV via Viva Insights web app or M365 Admin Center

Path A: CSV Org Data Upload (Full Capability)

Use when:
  • All agencies share the same email domain (e.g., everyone is @onetenant.gov.au)
  • You need both agency isolation AND department/location/job function filtering within each agency
  • You want the richest possible analytics (custom attributes, before/after cohort segmentation)
  • HR can provide a consolidated employee CSV with an Agency column

Partition filter: Agency = "Agency A"

End-to-end workflow:

  1. Phase 0: Pre-flight audit (same as Path B)
  2. Phase 1: Lockdown — disable AutoCxO, delegation, Agent Dashboard, Metrics Export (same as Path B)
  3. Phase 2: Prepare and upload CSV with PersonId, ManagerId, Organization, EffectiveDate, and custom Agency attribute. Verify 100% coverage on Data Quality page.
  4. Phase 3: Create partitions filtered by Agency = "Agency A", etc. Assign analysts.
  5. Phase 4: Analysts run Copilot PBI queries within their partition → publish reports.

Time estimate: 2–3 days. Ongoing: Monthly CSV re-upload to keep data current.

Path B: Entra ID Domain Filter (Zero CSV)

Use when:
  • Each agency has a distinct email domain (e.g., agencya.gov.au, agencyb.gov.au)
  • All employees use their agency domain as their primary SMTP address
  • You do not need custom attribute filtering (Department/Location come only from Entra defaults)
  • You want the fastest possible deployment with zero data preparation

Partition filter: Domain = "agencya.gov.au"

End-to-end workflow:

  1. Phase 0: Pre-flight audit (same as Path A)
  2. Phase 1: Lockdown — disable AutoCxO, delegation, Agent Dashboard, Metrics Export (same as Path A)
  3. Phase 2: Skip CSV upload. Optionally enable ManagerId and Organization in “Configure Entra connection” for richer hierarchy and department data.
  4. Phase 3: Create partitions filtered by Domain = "agencya.gov.au", etc. Assign analysts.
  5. Phase 4: Analysts run Copilot PBI queries within their partition → publish reports.

Time estimate: 1 day (no data prep). Ongoing: No CSV maintenance — Domain auto-updates as employees join/leave.

Decision Matrix

QuestionIf Yes → Path
Do all agencies share the same email domain?Path A (CSV required — Domain cannot differentiate agencies)
Does each agency have a unique email domain?Path B is available (can also use Path A for richer attributes)
Do you need Department/Location/Job Function filters per agency?Path A (CSV adds these attributes beyond Entra defaults)
Are there shared-services staff on a common domain?Path A preferred (assign Agency = "Shared Services" in CSV)
Need the fastest deployment with minimal data preparation?Path B (zero CSV, Domain-only filter)
Want to start with Path B and add CSV later?Path B now, Path A later — you can always upload CSV to add richer attributes. Use the parallel data ingestion feature to keep Entra as source for ManagerId/Organization while adding custom attributes via CSV.
Important: Whichever path you choose, the Phase 1 lockdown and partition-based isolation architecture are identical. The only difference is how employee data reaches Viva Insights and which attribute you filter partitions on.

4.2 Entra ID Attributes Reference for Viva Insights

Understanding exactly which Entra ID attributes reach Viva Insights is critical for choosing the right path. Per Microsoft Learn:

Attributes NOT Available from Entra ID

The following Entra ID user profile fields are not synced to Viva Insights. If you need them, you must upload via CSV:

Entra ID FieldStatus in Viva Insights

DivisionNot synced
JobTitleNot synced — use CSV FunctionType

Common Misconception: Many customers assume that because they have Company, Division, or extensionAttributes populated in Entra ID, these will automatically appear in Viva Insights. They will not. Only the three attributes listed above (PersonId, ManagerId, Organization) sync from Entra. Everything else requires a CSV upload.

Parallel Data Ingestion (Best of Both Worlds)

You can use both Entra ID and CSV simultaneously via the parallel data ingestion feature:

  • Let Entra continue providing ManagerId and Organization (always accurate, auto-refreshing)
  • Upload a CSV with only the additional attributes you need (e.g., Agency, Location, CostCentre)
  • Entra-sourced attributes take precedence for the fields you select; CSV provides the rest

This is ideal for organisations starting with Path B (Domain-based) who later want to add richer attributes without disrupting existing Entra-based data flows.

5. Implementation Runbook

0 Phase 0: Pre-Flight Audit

Before locking anything down, audit the current state. Disabling policies does not revoke existing access.

  • Audit VFAM policies (Admin UI): Navigate to admin.microsoft.com → Settings → Microsoft Viva → Viva Insights → Set up and management → Turn Advanced Insights on or off. Review the status of each feature: AutoCxoIdentification, CopilotDashboardDelegation, AgentDashboard, CopilotMetricsReport. Document the current state (On/Off, scope: Everyone or Specific users).
  • Alternative (PowerShell): Run Get-VivaModuleFeaturePolicy -ModuleId VivaInsights for a scripted audit of all policies.
  • Audit dashboard access list: Admin Center → Settings → Viva → Viva Insights → Manage user access for Copilot Dashboard. Check who currently has access.
  • Identify any agency CEOs or leaders who already have dashboard access (auto-enabled or manually added).
  • Verify the Entra ID ManagerID hierarchy — confirm each agency CEO reports to a different top-level node (not to each other).

1 Phase 1: Foundation (Lockdown)

Prevent agency CEOs from being auto-identified as senior leaders and getting tenant-wide access.

  • Disable Auto CxO Identification (Admin UI): Admin Center → Settings → Viva → Viva Insights → Turn Advanced Insights on or off → select Auto CxO Identification → set to Off for Everyone → Save.
    Alternative (PowerShell): Add-VivaModuleFeaturePolicy -ModuleId VivaInsights -FeatureId AutoCxoIdentification -Name "DisableAutoEnableForAllAgencies" -IsFeatureEnabled $false -Everyone
  • Remove existing auto-enabled leaders: Admin Center → Settings → Viva → Viva Insights → Manage user access for Copilot Dashboard → remove any agency leaders who were auto-identified.
  • Disable Dashboard Delegation (Admin UI): Same “Turn Advanced Insights on or off” page → select Copilot Dashboard Delegation → set to Off for Everyone → Save.
    Alternative (PowerShell): Add-VivaModuleFeaturePolicy -ModuleId VivaInsights -FeatureId CopilotDashboardDelegation -Name "DisableDelegationForTenant" -IsFeatureEnabled $false -Everyone
  • Restrict dashboard access to Central IT: Create Entra group CentralIT-CopilotDashboard-Access. In Manage user access for Copilot Dashboard → add this group → remove all individual agency users.
  • Disable Agent Dashboard (Admin UI): Same page → select Agent Dashboard → set to Off for Everyone → Save. Then create a group-scoped policy to re-enable for Central IT only.
    Alternative (PowerShell): Add-VivaModuleFeaturePolicy -ModuleId VivaInsights -FeatureId AgentDashboard -Name "RestrictAgentDashboard" -IsFeatureEnabled $false -Everyone
  • Disable Copilot Metrics Export (PowerShell only): This feature does not currently have an Admin UI toggle. Use PowerShell: Add-VivaModuleFeaturePolicy -ModuleId VivaInsights -FeatureId CopilotMetricsReport -Name "DisableMetricsExport" -IsFeatureEnabled $false -Everyone
After Phase 1 — Verification:
Confirm no agency leaders appear in the dashboard access list. Verify all four VFAM policies (AutoCxo, Delegation, AgentDashboard, MetricsExport) show as disabled. Have an agency CEO attempt to access insights.cloud.microsoft and confirm access is denied.

2 Phase 2: Organizational Data

Prepare the foundational data that enables partitioning. Your approach depends on which path you chose in Section 4.1.

Path A: CSV Org Data Upload

  • Create a consolidated CSV file containing ALL employees across ALL agencies.
  • Include the critical custom attribute: Agency (e.g., "Agency A", "Agency B").
  • Include required fields: EffectiveDate, PersonId, ManagerId, Organization.
  • Upload via Viva Insights web app or M365 Admin Center.
Path A Verification:
Check Viva Insights Admin → Data Quality to verify the Agency attribute has 100% coverage. Verify employee counts per agency match HR records.

Path B: Entra ID Domain (No CSV)

  • No CSV upload required. The Domain attribute is system-generated from each employee’s primary SMTP address.
  • Recommended: Navigate to Viva Insights → Organizational data → Configure Entra connection → enable ManagerId and Organization checkboxes → Save. This enriches your data with manager hierarchy and department info (takes ~24 hours to sync).
  • Verify domain mapping: confirm that each agency’s employees use a distinct email domain (e.g., agencya.gov.au, agencyb.gov.au).
Path B Verification:
In Viva Insights Admin → Organizational data, verify that Domain appears as an available attribute. Check the Data Quality page to see Domain values and confirm they cleanly map to agencies. If any employees use a shared domain (e.g., contractors on shared.gov.au), decide how to handle them (see Shared Services Decision below).
Shared Services Decision (both paths): If your organisation has shared-services teams (IT, Legal, Finance) that serve multiple agencies, choose one of these approaches:
  1. Dedicated “Shared Services” partition (recommended) — Create a separate partition filtered by Agency = "Shared Services" (Path A) or Domain = "shared.gov.au" (Path B). Central IT analysts access this partition alongside the global partition.
  2. Global partition only — Exclude shared services from all agency partitions. Their data is only visible in the global partition (Central IT only).
  3. Include in every agency partition — Not recommended. Creates duplicate data and inflates agency metrics.

3 Phase 3: Partitions Setup

Create the hard data boundaries. The partition filter differs by path, but all other steps are identical.

Before Enabling Partitions — Confirm:
(a) Path A: You have a reliable, repeatable process for monthly org data uploads with the Agency attribute. Path B: You have verified Domain values cleanly map to agencies.
(b) You have identified at least one Insights Analyst per agency.
(c) You accept that once enabled, turning off partitions requires contacting Microsoft support and will delete all partition configurations, query results, and custom metrics created within partitions.
(d) You understand that privacy settings (minimum group size) are tenant-wide, not per-partition — all agencies must agree on the number.
  • Navigate to insights.cloud.microsoft → Admin → Privacy settings → Partitions. Select Enable Partitions.
  • Select New partition. Name it (e.g., “Agency A”).
    • Path A filter: Set Agency = "Agency A"
    • Path B filter: Set Domain = "agencya.gov.au"
    Verify the Filtered employees counter matches the expected headcount for that agency. Select Save.
  • Click the newly created partition → Edit → Analysts tab → Add analysts. Search for the designated agency analyst by name or email → select → Save. Assign agency analysts to their respective agency partitions only. Do NOT assign agency analysts to the global partition.
  • Assign Central IT analysts to the automatically created Global Partition for tenant-wide queries.
  • Repeat for each agency. Allow 1–2 hours for partition access to propagate.
Why agency analysts must NOT have global partition access: Analysts with global partition access automatically get Copilot Dashboard access AND Copilot Analytics Reports access — both of which show tenant-wide data across all agencies. This would defeat the entire isolation architecture.
After Phase 3 — Verification:
Have each agency analyst sign in to insights.cloud.microsoft. Confirm they only see their agency’s partition in the partition switcher. Have them run a test person query and verify the employee count matches their agency only.

4 Phase 4: Copilot Reporting

Generate and distribute the isolated reports.

  • Agency analysts run Copilot Power BI template queries within their isolated partition.

Option A: Power BI Workspaces (requires Power BI Pro/Premium licences for viewers)

  • Create dedicated Power BI workspaces per agency (e.g., Agency-A-Copilot-Analytics).
  • Publish reports and grant Viewer access to agency leadership via Entra group.
  • Configure Row-Level Security (RLS) as a secondary control if multiple agencies share a workspace.

Option B: Viva Insights Publish Reports (no Power BI licence required for viewers)

  • Use the Publish Reports feature in the Viva Insights analyst experience.
  • Select “Filter report by manager hierarchy” to automatically scope data to the recipient’s team.
  • Recipients view published reports in the Viva Insights web app → Reports tab.
  • Set expiry date (up to 365 days) and republish as needed.
After Phase 4 — Verification:
Have an agency CEO view their report (either PBI or Viva Insights). Confirm: (a) only their agency’s employees appear, (b) no cross-agency group names or data are visible, (c) the minimum group size threshold is respected for small departments within the agency.

6. Organizational Data Strategy

How employee data reaches Viva Insights depends on your chosen path. Both paths share the same Copilot usage data (automatic from M365).

6.1 Path A: CSV Org Data Upload

Data Type Source Refresh Cadence
Employee hierarchy (ManagerId) Centralized HR or per-agency HR feeds (CSV) Monthly (minimum)
Custom attributes (Agency, FunctionType) HR system (via CSV) Monthly
Copilot usage data Automatic from M365 signals Daily (6-day delay)

Data Quality Rule: The Agency attribute must have 100% coverage. Missing values will exclude employees from all partitions.

Data Refresh Failure Alerting (Path A):
If a monthly org data upload fails or the Agency attribute is missing for new employees, those employees silently disappear from all partitions. Nobody gets alerted.

Mitigations:
  • After each upload, check the Data Quality page in Viva Insights Admin to verify attribute coverage.
  • Set a calendar reminder to verify Agency attribute has 100% coverage within 7 days of each upload.
  • Consider automating uploads via the Viva Insights API-based import or Azure blob import for more reliable cadence.

6.2 Path B: Entra ID Domain (No CSV)

Data Type Source Refresh Cadence
Domain (agency identifier) Primary SMTP address (system-generated) Automatic — updates as employees are added/removed in Entra
Employee hierarchy (ManagerId) Entra ID (enable via Configure Entra connection) Automatic — ~24-hour sync
Department (Organization) Entra ID Department field (enable via Configure Entra connection) Automatic — ~24-hour sync
Copilot usage data Automatic from M365 signals Daily (6-day delay)

Key advantage: No ongoing data maintenance. New employees automatically appear in the correct partition based on their SMTP domain. Departing employees are automatically removed when their account is disabled.

Path B Limitation: You cannot filter by custom attributes (Cost Centre, Job Function, Location, etc.) within agency partitions. If you need these filters, upload a CSV using the parallel data ingestion method — Entra continues providing ManagerId and Organization while the CSV adds custom attributes.

6.3 Upgrading from Path B to Path A

Organisations can start with Path B for rapid deployment and later add a CSV upload to unlock richer analytics:

  1. Enable parallel data ingestion in Viva Insights → Organizational data → Configure Entra connection.
  2. Keep Entra as the source for ManagerId and Organization.
  3. Upload a CSV with additional attributes (Agency, FunctionType, Location, etc.).
  4. Update partition filters from Domain to Agency if preferred (edit each partition → change filter condition).

8. Governance & Ongoing Operations

  • Monthly (Path A only): Upload updated organizational data CSV (Central IT). Path B requires no recurring data maintenance.
  • Monthly: Refresh Copilot PBI template queries and publish reports (Agency Analysts).
  • Quarterly: Audit dashboard access list and partition memberships.
  • Onboarding: To add a new agency: Path A — add them to the CSV, create a new partition, assign an analyst. Path B — create a partition filtered by the new agency’s domain, assign an analyst. Both paths: spin up a new PBI workspace or use Publish Reports.

10. Discovery Questions for Customer

  1. Does each agency have a distinct email domain, or do all agencies share one domain? (Determines whether Path B — Domain-based isolation — is viable, or whether Path A — CSV with custom Agency attribute — is required. See Section 4.1).
  2. How is the Entra ID ManagerID hierarchy currently structured? (Determines if CEOs are auto-identified as senior leaders).
  3. How many total employees per agency, and how many have Copilot licenses? (Partitions need sufficient population to meet the minimum group size threshold, default 10).
  4. Are there shared-services teams (IT, Legal, Finance) that serve multiple agencies? (Determines if a "Shared Services" partition is needed).
  5. What minimum privacy threshold (minimum group size) is acceptable? (This is set tenant-wide; all agencies must agree on the number).
  6. Do agency leaders need Power BI Pro/Premium licences, or should we use Viva Insights Publish Reports? (Determines report distribution method — Phase 4 options.)
  7. Has anyone already been granted Copilot Dashboard access or does any analyst have global partition access? (Must audit and remediate before proceeding with Phase 1 lockdown.)
  8. Is there an automated HR data feed, or will org data uploads be manual CSV? (Impacts reliability of partition boundaries over time.)
  9. Are Copilot agents deployed across agencies? If so, does agent usage data also need isolation? (Determines whether Agent Dashboard needs VFAM lockdown.)
