Why Multi-Agency Analytics Is Hard
The architectural challenge of scoping analytics across agencies in a shared tenant
Document PropertyValue
SectionExplain

When This Applies

Multi-agency analytics is a challenge when your organisation meets all of the following conditions:

  • Single M365 tenant — all agencies share one tenant
  • Multiple independent agencies (or business units, divisions, departments) that operate semi-autonomously
  • Each agency needs Copilot analytics scoped to their own population

The fundamental tension: Microsoft 365 Copilot analytics has several separately governed surfaces. In a shared tenant, there is no single partition or product setting that scopes the Copilot Dashboard, Agent Dashboard, metrics export, delegation, manager access, analyst queries, privileged roles, and downstream Power BI sharing together. A user with a tenant-wide access source can see “Your company” data even when an agency analyst is assigned only to an agency partition.

The patterns below coordinate those independent controls. They reduce accidental cross-agency reporting, but they do not create cryptographic, tenant, or product security isolation.

Single-Agency Tenants

If your tenant serves only one agency (or your agencies don’t need separate analytics scoping), none of this applies. Follow the standard Admin Setup Guide instead.

The Two Architectures

THIS GUIDE’S AUTHOR-CREATED ARCHITECTURE PATTERNS

Default Priority and Restricted Reporting (formerly labelled “Isolation” in this guide and retained in existing URLs) are editorial names created by this guide. They are not Microsoft product, security, compliance, or licensing terms. Microsoft documents the component capabilities, including partitions; Microsoft does not document this guide’s combined pattern as an end-to-end security boundary.

Default Priority is the simpler author-created option. An agency leader’s default dashboard scope can follow their Entra ID manager hierarchy when Microsoft’s access and hierarchy conditions are met. The trade-off is intentional: a user with a tenant-wide access source may still use “Your company”, and delegates inherit their delegator’s scope. Choose this only when that visibility is accepted and documented. Setup typically takes 2–4 hours.

Restricted Reporting is the more controlled author-created option. It combines group-scoped Viva feature access management (VFAM), partition-scoped analyst query workspaces, restricted global access, and dedicated Power BI delivery. A partition limits the rows and columns available to an assigned analyst in that partition’s query workspace; it does not govern every other Microsoft 365, Viva Insights, export, Agent, delegation, manager, admin, or Power BI surface. Treat the pattern as defense in depth and reporting segmentation. A regulatory, contractual, sovereign-cloud, or security requirement still needs a tenant-specific control assessment, Microsoft Support validation where behavior is unclear, and Compliance/Privacy/DPO approval. Setup typically takes 1–3 days.

Within Restricted Reporting, there are two data paths depending on whether agencies share an email domain:

  • Path A — shared domain (e.g., all users are @government.gov). Requires organizational data with a custom Agency attribute to distinguish agencies. Refresh it whenever the source hierarchy changes.
  • Path B — distinct domains (e.g., @agencyA.gov, @agencyB.gov). Uses the system-generated Domain attribute from Entra ID. No CSV needed for partitioning, though you may still want one for additional filter attributes.
Organizational Data Precedence Is Attribute- and Surface-Specific

Do not describe a file upload as universally or permanently replacing Entra data. Uploaded values change the source for supplied dashboard attributes; parallel ingestion can keep or restore Entra for ManagerId and Organization, while Microsoft 365 Profile has separate precedence. Before every change, record the current source for each required attribute in Data Quality/Profile, check Setup → Migration and imports plus Message Center, and use the upload path actually supported in the tenant. Microsoft’s upload articles currently describe a tenant-wave transition and differing legacy behavior.

Supported-Environment Gate

Stop Before Design If the Required Surface Is Unsupported

As of the dated Microsoft sources below, the Agent Dashboard is preview and its feature-specific article uses the conservative prerequisite of at least 50 Microsoft 365 Copilot licences plus agent activity; a tenant with only 50 Viva Insights licences must validate availability. Agent Dashboard is unavailable in national and regional clouds. Copilot metrics export is preview, and Microsoft’s supported-environments matrix says GCC does not support Agent Dashboard, Copilot metrics export, or Copilot Power BI reports. Base Viva plan support does not establish feature parity. Validate the exact cloud, tenant UI, licences/service plans, Message Center, and required surface before implementation; escalate a documented mismatch to Microsoft Support.

Mandatory Go-Live Evidence Gate

  1. Test a representative account for every actor: agency leader, cross-agency manager, shared executive, delegate, agency analyst, multi-partition analyst, global-partition analyst, report Viewer, Power BI Member/Contributor, AI Administrator, Insights Administrator, Insights Analyst, and Global Administrator.
  2. For each account, run negative tests against the Copilot Dashboard, Analytics Reports, Agent Dashboard, weekly/daily metrics export, advanced query workspace, query-result download, delegation, manager-published reports, Power BI workspace/app, report export, Analyze in Excel, and resharing.
  3. Capture dated evidence: tester identity/role, group and partition memberships, VFAM/access policy export, org-data source and quality snapshot, expected denial, observed result, screenshot or audit record, Power BI permissions, test owner, and remediation ticket.
  4. Publication and go-live are vetoed if any negative test fails or cannot be executed. Remove access or roll back the change, preserve evidence, and open Microsoft Support when documented behavior and tenant behavior differ.
  5. Obtain written Security, Compliance/Legal, Privacy/DPO, Viva Insights owner, and Power BI owner sign-off on the tested behavior and residual risks. Product documentation alone is not an attestation.
  6. Repeat the full test quarterly and after any role, manager hierarchy, delegation, org-data, partition, VFAM, licence, cloud, preview-status, or Power BI sharing change.

Rollback and Change Control

Record the current policies, access lists, role assignments, partition filters/analysts, org-data source/quality, query assets, and Power BI permissions before change. Use a test cohort first. Treat enabling partitions as a high-impact tenant change; Microsoft documents Support involvement to turn partitions off, and deletion can remove partition configuration and associated assets. For org-data changes, use supported update/delete behavior rather than dummy rows. Define an owner, maintenance window, success criteria, failure criteria, restoration sequence, approvers, and Support trigger before execution.

Ready to Decide?

If cross-agency dashboard visibility is accepted, evaluate Default Priority. If the organisation needs stronger reporting segmentation, evaluate Restricted Reporting, then determine the data path based on whether agencies share an email domain. If the requirement is cryptographic, tenant, or independently certified security isolation, neither author-created pattern establishes it; use a security architecture assessment to select an appropriate boundary.
